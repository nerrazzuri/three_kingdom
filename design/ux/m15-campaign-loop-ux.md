## 0. 背景：为什么需要这一层

M00~M10 已建成完整的 11 循环（`CampaignSession`，797 测试证明可跑）。M15 交付物① 控制台 harness 让它**首次有人类入口**。但「能操作」≠「能理解」。module-plan 给 M15 的职责是一句话：

> **让玩家知道自己为什么能赢、为什么会败、下一步还能做什么。**

本作是**不完全信息的回合制策略沙盒**，核心幻想是「我赢因为我创造了条件，而非点了技能」。这要求表现层做到三件别的策略游戏常做反的事（沿用 game-pillars）：
- **别替玩家判断**（P11 无最优解/无自动捷径）——军师给缘由不给胜率，系统不替你布阵。
- **别泄露不该知道的真值**（P10 反全知）——敌情只呈估计值 + 来源 + 时效。
- **别把多维状态压成一个数字**（P6）——民心/治安/工事/士气/疲劳各自成维。

M15 的成败，不在画面华丽，而在玩家**每一步都能复盘因果、看懂风险、知道还能做什么**。

## 1. 全循环信息架构（Loop-to-Loop Flow）

11 个循环不是平铺的菜单项，而是一条有自然顺序的太守日常。玩家心智模型应是：**察 → 谋 → 备 → 战 → 果 → 长**，套在「世界时间在我推进时才前进」（P8）之上。

```
            ┌─────────────────────── 世界时间轴（仅玩家推进时前进，P8）───────────────────────┐
            │                                                                                  │
  [开局] ──▶ 察(情报/侦察) ──▶ 谋(军议) ──▶ 备(战役准备/设伏) ──▶ 战(兵法战斗) ──▶ 果(战果/续局)
            │       ▲                                                                   │
            │       └──────────── 治理(城市) 持续供养：粮/民心/工事 喂给战与备 ──────────┘
            │                                                                           │
            └──▶ 长(生涯：记功/晋升/自立) ◀── 果 ── 推进时段 ──▶ 历史世界(够不着继续/够得着分叉)
```

**信息架构分级**（哪些常驻、哪些情境浮现、哪些按需、哪些永不显示）——延续 `hud.md` §2 的四级法则，扩展到循环层：

| 循环 | 玩家此刻必须读懂的一句话 | 主反馈契约（见 §2） | 永不显示（红线） |
|---|---|---|---|
| 开局 | 我是谁、守哪座城、面对谁、目标是什么 | — | — |
| 城市治理 | 这座城明天还能不能供养我打仗 | 因果（治理→战役条件） | — |
| 情报/侦察 | 我对敌的了解有多旧、多可信 | 情报置信/时效 | 敌方精确真值 |
| 军议 | 军师指出了哪些可能、缺什么情报、风险是什么 | 风险（无胜率） | 最优解/胜率/自动布阵 |
| 战役准备 | 我投入了什么、是否够、提交后不可反悔 | 风险 + 承诺（草稿 vs 已提交） | — |
| 兵法战斗 | 我满足了哪些条件、还差什么才成势 | 因果（条件涌现） | 「点一下必胜」的按钮 |
| 战果/续局 | 我为什么赢/败、现在还能做什么 | 因果 + 续局 | 「游戏结束/删档」（失败必可继续） |
| 生涯 | 我离晋升/自立还差什么、非战斗路也算数 | 风险（门槛可解释） | 「唯一最优晋升路」 |
| 历史世界 | 我够不够得着改变历史、我改了什么 | 因果（够不着短路/够得着分叉） | 上帝视角全局真相 |

> **MVP 信息架构原则**：任一循环，玩家若**看不到「为什么」和「下一步」**，就是 M15 的缺陷。热闹但不解释因果的元素一律退为按需。

## 2. 四个跨循环反馈契约（M15 的权威产出）

这是 M15 真正拥有的东西——不是任何单个循环的规则，而是**贯穿所有循环的四类反馈如何呈现**。控制台 harness 已落了每一个的最小版（括注其参考实现），Unity UI 须满足同样契约。

### 2.1 因果契约（Causal Chain）
**承诺**：每一个结果都能追溯到「玩家做了什么 + 世界条件如何 → 故有此果」，且 ≤5 个主因素（沿用 BattleOutcomeSummary 的 CausalTrace 约定）。
- 战斗：复盘列出**满足了哪些兵法条件**、识别出哪条成型兵法（harness：`[t]识别兵法` → 「假退伏击（满足条件 3 条）」）。
- 治理：每个治理动作显示它**如何改变后续战役条件**（修工事↑守城强度 / 征用↑补给↓民心）。
- 历史：分叉时明示「你已强到改变前置，历史脱稿」。
- **反例（禁止）**：只显示「胜/败」不显示因果链。

### 2.2 风险契约（Risk without Win-Rate）
**承诺**：呈现「条件、风险、缺失情报、定性置信」，**绝不**给成功率数字或唯一推荐（P11 / GDD_008 设计锁）。
- 军议（harness `[6]`）：缘由 + 所需条件 + 风险 + 待证情报 + 完整度/置信（0.00–1.00 表「军师把握的完整度」，**非胜率**）。
- 战役准备：提交前显示投入与硬约束；不预测结果。
- **反例（禁止）**：「此计成功率 73%」「推荐方案：设伏」。

### 2.3 情报置信/时效契约（Confidence & Staleness）
**承诺**：敌情永远以**估计值 + 来源 + 观测时效**呈现，会过期，且不静默更新已发出的军议（反全知 P10）。
- 侦察（harness `[5]` → 状态行 `subject-enemy-army≈5000（Scouting，N 段前）`）：估值 + 来源 + 「N 段前」时效。
- 军议绑定召开时的知识快照；之后侦察改变知识 → 旧建议标记为「过时」（`IsStaleAgainst`）。
- **反例（禁止）**：显示敌军精确实时兵力；侦察后旧建议自动刷新。

### 2.4 续局契约（Continuation after Outcome）
**承诺**：任何战果——尤其失败——都给出 ≥1 个**合法可继续选项**，绝无「游戏结束/删档」（control-manifest 强制设计锁「失败可继续」+ 终盘决策一「失败不删档」）。
- 战果结算（harness `[o]胜 / [p]败`）：四分支（胜/撤/失城/败北）各有续局选项；失城仍「势力延续」、罢官转「在野亦为合法续局」。
- **反例（禁止）**：败局弹「Game Over」要求读档重来。

## 3. 新手循环序（Onboarding）

**范围**（module-plan：M15 是 onboarding/可读性/反馈，**非仅**首小时教程；完整难度矩阵归 M16）。原则：**沿自然循环序、用 in-world 缘由教学，不做模态弹窗串讲**。

第一局「汜水关守御」即教学雏形（沿用竖切场景）。建议的渐进暴露：

1. **先察后谋**（情报→军议）：开局敌情未明，引导玩家先侦察；军议自动呈现（harness 前 3 回合自动显示军师建议的设计已在竖切验证）。
2. **谋而后备**（军议→准备）：军师点出「隘口宜设伏」缘由后，引导玩家自己组准备计划——**不替选**（教「投入谁/多少/借何势」是玩家的事）。
3. **备而后战**（准备→战斗）：提交计划不可反悔的承诺感，用草稿 vs 已提交的视觉区分（朱批朱红，见 hud.md §4）。
4. **战中见条件**（战斗）：让玩家看到兵法是**逐条满足条件涌现**的（harness 的「还差 N 条」即此教学的文本原型），而非按钮。
5. **果后知续**（战果）：无论胜败，明确「下一步还能做什么」。**首次失败**尤其要强调「可继续」，扭转玩家「输了=重来」的预期。
6. **长线建立**（生涯/历史）：晋升门槛、非战斗功绩也算数、历史在轨上跑——这些在第一局后逐步引入。

> **Open Question（待实玩）**：上述序在真实游玩中哪一步最易卡手？harness 实玩反馈将填此空。

## 4. 渲染与交互的硬约束（对 console harness 与 Unity UI 同时生效）

延续 coding-standards / module-plan §M15 测试证据：

1. **只读投影 + 只提交 Command**：表现层**不引用 Domain 内部可变状态**，不直接改 state；一切操作经 Application 用例（ADR-0002/0009）。harness 已守（`BattleConditions`/`HistoryCatalog` 等 internal 成员跨程序集不可见，编译期挡住）。
2. **反全知在类型层兜底**：情报只读「玩家阵营知识」出口（`PlayerKnowledge`），真值 internal，表现层够不着。
3. **确定性可复现**：同会话态渲染恒等（纯函数）；同输入序列同结果（harness 已单测 `test_scripted_play_is_deterministic`）。
4. **多维不压扁**：民心/治安/工事/士气/疲劳分列显示，不合并为单一「实力」数字（P6）。
5. **无障碍**：对齐 Comprehensive 层——色盲不靠纯色区分（形状/文字冗余）、文本缩放、减少动态、键鼠全可达、无仅悬停可触发状态（technical-preferences 平台说明）。

## 5. Tuning Knobs（M15 可配置项）

| Knob | 说明 | 默认（harness） |
|---|---|---|
| 军议自动显示回合数 | 新手前 N 回合自动展开军师建议，之后按 `?` 调出 | 3（竖切沿用） |
| 因果链展开深度 | 复盘默认展开 vs 折叠；最多主因素数 | ≤5 因素 |
| 情报时效告警阈值 | 知识「N 段前」超过阈值高亮为过时 | 与 IntelConfig.ttlSegments 一致（8） |
| 置信显示粒度 | 定性（高/中/低）vs 小数 | harness 暂用小数 0.00–1.00（待评审是否改定性档） |

## 6. Acceptance Criteria（可玩性验收）

- **AC-1 因果可读**：每个循环的结果都能展示 ≤5 主因素的因果链；战斗复盘能列出满足的兵法条件与识别出的成型兵法。
- **AC-2 无胜率泄露**：全游戏任何界面**不出现**成功率数字或「推荐最优方案」；军议只给缘由/条件/风险/置信。
- **AC-3 反全知**：敌情仅以估计值+来源+时效呈现；不存在显示敌方精确真值的路径（类型层可证）。
- **AC-4 失败可继续**：四类战果分支每一类都呈现 ≥1 续局选项；无任何「游戏结束/删档」终点（单测 `test_defeat_outcome_is_continuable` 已覆盖 harness）。
- **AC-5 下一步明确**：任一循环态下，玩家都能看到「现在可做的合法动作」（harness 的命令菜单即最小版）。
- **AC-6 确定性渲染**：同会话态渲染恒等；表现层不引用 Domain 可变内部态（编译期 + 单测可证）。
- **AC-7 无障碍**：达成 Comprehensive 层关键项（色盲冗余编码、文本缩放、键鼠全可达）。

> **验收范围说明**：AC-1~AC-6 在 console harness 上已**部分或全部**可证（见各处 harness 注）。Unity UI 实现时须在引擎面重新满足同一组 AC（M15 交付物②，归 epic-028 后续 story）。

## 7. Open Questions（待用户实玩 / 评审）

1. **置信显示**：军师「完整度/置信」用小数（0.56）还是定性档（高/中/低）？小数易被误读为胜率——倾向改定性，待你实玩判断。
2. **onboarding 卡点**：§3 哪一步真实游玩最易卡手？（需 harness 实玩反馈）
3. **因果链默认展开 vs 折叠**：复盘信息量大，默认全展开会否淹没新手？
4. **console harness 的定位**：仅作内部验证/盘点工具，还是保留为「文本模式」可玩形态（无障碍/低配/快速验证价值）？影响是否给它投入打磨。
5. **Unity UI 切入点**：交付物② 优先接哪一屏？（建议从「战果复盘」起——因果契约最吃 UI，且最能验证 M15 核心幻想）。
