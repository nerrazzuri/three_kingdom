# GDD_011 — 士气与疲劳

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

表达部队继续行动的意愿与身体负荷，使胜负受到经历、补给、休息、指挥和军纪影响，并为断粮、夜袭、假退与持久守城提供延迟后果。

## Player Fantasy

玩家能观察部队为何振奋、动摇或精疲力竭，通过休整、补给、可靠指挥和谨慎用兵保持战斗力，而不是管理两条无来源的数值条。

## Core Loop

查看士气、疲劳、军纪及近期原因 → 安排行动、休整与补给 → 战役/时段事件累积变化 → 状态带影响命令可靠性和风险 → 玩家调整节奏 → 结果继续写入原因账本。

## Main Rules

- 士气表示继续战斗的集体意愿；疲劳表示身体与组织负荷；军纪表示在压力下维持命令和队形的能力，三者不得合并。
- 状态保存连续权威值并映射为可读等级；阈值和曲线来自配置。
- 士气变化由具名事件产生，如伤亡、侧翼受威胁、指挥者失能、成功撤退、补给恢复或援军消息。
- 疲劳随移动、交战、夜间活动、恶劣天气、缺粮和负载累积；通过安全休整、食物和时间恢复。
- 军纪在 MVP 中是较慢变化状态，影响受控撤退、隐蔽待命、夜间协同与溃散抵抗；不由单次小事件任意大幅变化。
- 状态在战役阶段或世界时段边界统一结算，不依赖动画时长。
- 到达阈值触发动摇、混乱、溃散等检查，但结果同时考虑军纪、指挥和退路；单一数值不自动决定全部行为。
- 将领影响通过具体命令、在场、关系与能力因素传导，不提供永久全图士气光环。

## Formulas

> 士气（继续战斗的意愿）、疲劳（身体/组织负荷）、军纪（压力下维持队形的能力）
> **三者独立、不得合并**。本系统 `morale` 指**部队士气 `unit_morale`**，与 GDD_004 城市民心 `civ_morale`
> 是不同权威系统的不同状态，配置/代码不得混用同名（见 GDD_004 §4 命名消歧）。
> 状态保存连续权威值并映射为可读等级。阈值检查同时考虑
> 军纪、指挥与退路——**单一数值不自动决定全部行为**。指挥者**不**提供永久全图士气光环。
> 同一事件不重复结算（幂等）。所有强度、曲线、阈值来自版本化配置。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `morale`、`fatigue`、`discipline` | 三维连续状态 | [0, 1] | CohesionState |
| `Δm_event` | 士气事件强度 | 实数 | MoraleEvent |
| `audience_w` | 事件对该单位的受众权重 | [0, 1] | 配置 |
| `fat_gain(action)` | 行动疲劳积累量 | ≥ 0 | 配置 |
| `rest_rate` | 安全休整恢复率 | [0, 1]/时段 | 配置 |
| `cmd_infl` | 指挥者影响（在场/能力/关系） | 有上限 | GDD_005/006 |
| `level_thresholds` | 连续值→等级的阈值表 | 配置 | 配置 |

### 1. 士气变化（具名事件聚合，幂等）

同阶段多事件按受众权重聚合后一次应用,指挥影响有上限:

```
morale' = clamp( morale + Σ_e audience_w[e] × Δm_event[e] + min(cmd_infl, cmd_cap), 0, 1 )
```

- 指挥者失能与胜利消息同阶段发生时,按事件排序与聚合规则共同结算(见 §Edge Cases)。
- **示例**：morale=0.6，伤亡 Δ=−0.2（权重 1.0），援军消息 Δ=+0.1（权重 0.8），cmd_infl=0.05(cap 0.1) → `0.6 + (−0.2 + 0.08) + 0.05 = 0.6−0.12+0.05 = 0.53`。

### 2. 疲劳积累与恢复

```
fatigue' = clamp( fatigue + Σ_a fat_gain(a) × env_mult − rest_rate × is_resting, 0, 1 )
```

- 移动/交战/夜间/恶劣天气/缺粮/负载增加积累;安全休整 + 食物 + 时间恢复。休整区被接触则中断,保留已恢复部分(见 §Edge Cases)。
- **示例**：fatigue=0.3，夜行 fat_gain=0.15，env_mult=1.2（雨）→ 积累 `0.15×1.2=0.18`；未休整 → `0.3+0.18=0.48`。
- **断粮传导单一权威**：缺粮对 morale/fatigue 的影响**仅在本系统**（§1 士气、§2 疲劳）施加，由 GDD_012 §5 的断粮后果事件触发（幂等）；GDD_010 只读取本系统已结算值，不自行再减——杜绝同一饥饿惩罚被重复计算。

### 3. 军纪（缓慢变化）

```
discipline' = discipline + slow_rate × (Δdisc_event)   其中 |slow_rate × Δdisc_event| ≤ disc_max_step
```

- MVP 中军纪缓慢变化,不由单次小事件大幅跳变(见 §Main Rules)。主要作为初始与后果状态(见 §Open Questions)。

### 4. 阈值检查（多因素，非单值决定）

到达士气阈值触发动摇/溃散**检查**,但结果综合军纪、指挥、退路:

```
rout_check = ( morale < morale_floor )
           ∧ ( discipline × command_quality × route_clear < hold_threshold )
           ∧ ( r < rout_prob_curve(morale, discipline) )
```

- 三重条件 + 检查点随机流;高军纪/有退路的低士气单位可有序撤退而非溃散(见 §Edge Cases)。
- **示例**：morale=0.2<floor 0.3，discipline×cmd×route=0.5×0.8×1.0=0.4 ≥ hold 0.35 → 第二条件不成立 → 不溃散,转受控撤退。

### 5. 拆分/合并状态组合（按人数加权，非取最大）

```
state_merged = Σ_i ( headcount_i × state_i ) / Σ_i headcount_i   （军纪另按军纪规则微调）
```

- 单位拆分/合并时三维状态按人数与经历加权,不能简单取 max(见 §Edge Cases)。
- **示例**：单位 A（1000 人，morale 0.7）+ B（500 人，morale 0.4）→ `(1000×0.7+500×0.4)/1500 = (700+200)/1500 = 0.6`。

### 6. 连续值→可读等级映射

```
level(value) = 最大的 L 使得 value ≥ level_thresholds[L]
```

- 玩家见等级与趋势(精确值是否可见见 §Open Questions);敌军状态只显情报推测。变化动画可跳过,不影响结算(见 §UI)。

## Data Model

- `CohesionState`：单位 ID、士气值/等级、疲劳值/等级、军纪值/等级。
- `MoraleEvent`：来源、强度、受众、可见性、时间、持续/衰减标签。
- `FatigueEvent`：行动、负载、环境、恢复条件和变化量。
- `DisciplineCheck`：触发、因素、随机流引用、结果与原因。
- `CohesionModifier`：具名来源、期限、影响维度。
- `CohesionProjection`：玩家可见等级、趋势和主要公开原因。

## Player Inputs

命令休整、轮换、补给、撤退或继续行动；选择指挥者和风险策略。玩家不能直接“鼓舞 +X”或编辑状态；讲话/奖赏若未来存在也必须消耗资源并产生关系后果。

## System Outputs

状态值与可读等级、趋势、原因事件、命令风险修正、受控撤退/追击/隐蔽协同检查、溃散或恢复事件，以及战后单位与人物后果。

## Dependencies

依赖 GDD_001 时间、002 环境、005 指挥者、006 关系、010 战役事件和012 补给。向战前准备、战斗 AI 与后果结算提供状态。亦被 GDD_016（敌方 AI 读自身士气/疲劳态势作效用评分）消费。

## Edge Cases

- 高士气但高疲劳的部队愿战却执行迟缓；低士气但高军纪的部队可能有序撤退。
- 指挥者失能与胜利消息同阶段发生时，按事件排序和聚合规则共同结算。
- 单位拆分/合并时状态按人数、经历与军纪规则组合，不能简单取最大值。
- 休整区域被敌军接触时中断恢复并保留已完成部分。

## Failure Cases

状态越界、未知事件、重复事件、无效单位或同一事件重复应用必须拒绝且不改状态。溃散、拒绝追击或无法受控撤退是 gameplay 结果，并提供决定性因素。

## Balancing Parameters

状态范围与等级阈值、事件强度、衰减、疲劳积累/恢复、夜间/天气/缺粮修正、军纪检查曲线、指挥影响上限、溃散与重整条件。全部来自版本化配置。

## UI Requirements

分别显示士气、疲劳和军纪的等级、趋势及主要原因；提交命令时提示相关风险因素而非单一成功率；敌军状态只能显示情报推测；变化动画可跳过且不影响结算。

## AI Requirements

AI 读取自身单位真值和敌方已知估计，遵守相同恢复与崩溃规则。性格影响其愿意承受的状态风险，不允许 AI 免疫疲劳、断粮或溃散。

## Save / Load Requirements

保存权威状态、未结算事件、临时修正、恢复进度、检查随机流和公开原因历史。加载后不得重复结算阶段事件。

## Test Requirements

测试三维状态独立性、事件聚合、阈值、越界、重复事件、休整中断、缺粮延迟、指挥者失能、受控撤退、拆分合并、AI 同规则和重放一致。

## MVP Scope

单位级士气、疲劳和军纪；有限事件来源、休整、缺粮、夜间行动、指挥者失能、动摇与溃散。无复杂心理疾病或长期训练体系。

## Future Scope

老兵/新兵差异、创伤、文化、长期训练、军饷、宣传、家庭消息、单位传统、俘虏冲击和更细的组织凝聚力。

## Open Questions

- 军纪在 slice 内能否通过短期训练显著改变，还是主要作为初始与后果状态？
- 玩家看到己方精确值还是只看等级与趋势？
- 单位溃散后重整由玩家命令、自动条件还是两者结合？
