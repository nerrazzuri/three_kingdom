# GDD_006 — 关系与派系

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

表达人物之间的信任、敬重、恩义、怨恨与组织立场，使授权、情报共享、协作意愿和战后问责具有社会原因。

## Player Fantasy

玩家通过长期行为赢得或失去他人支持；同一道命令由谁提出、谁执行、会伤害谁的利益，都会改变方案可行性。

## Core Loop

查看已知关系和职责 → 请求授权或协作 → 人物基于关系、职责、性格与风险响应 → 行动和结果产生关系事件 → 更新后续信息共享、执行意愿和政治后果。

## Main Rules

- 关系不是单一好感度；MVP 记录信任、敬重、恩义与怨恨四个方向性维度。
- `A 对 B` 与 `B 对 A` 分别保存，不要求对称。
- 正式职责提供基础权限，关系只能影响请求、支持和执行承诺，不能凭空授予法律权限。
- 协作评估输出接受、反对、要求保证或拒绝等结构化结果及可公开原因。
- 一旦人物明确接受任务，执行偏差来自能力、环境与已知风险；关系不会每阶段随机翻转命令。
- 关系变化必须由具名事件驱动，并受知情范围影响；未被知晓的行为不立即改变所有人关系。
- MVP 派系只表达城市内有限立场与问责，不模拟完整朝堂。

## Formulas

> 关系是方向性、多维的：`A对B` 与 `B对A` 分别保存，不要求对称。关系变化必须由
> 具名事件驱动且受知情范围影响——未被知晓的行为不改变关系。关系**只影响**请求/支持/
> 执行承诺，**不能**凭空授予法律权限（权限来自 GDD_005 职责）。所有变化量、权重来自配置。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `rel[A→B][dim]` | A 对 B 在维度 dim 的值（信任/敬重/恩义/怨恨） | DIM_MIN..DIM_MAX | RelationshipState |
| `Δevent[dim]` | 事件对某维度的变化量 | 实数 | 配置（事件档案） |
| `knows(c, event)` | 人物 c 是否知情该事件 | 布尔 | RelationshipEvent.知情者 |
| `w_coop[dim]` | 协作评估对维度 dim 的权重 | 实数 | 配置 |
| `mem_decay[dim]` | 各维度记忆衰减率 | [0, 1] | 配置 |
| `risk_pen` | 任务已知风险对协作的惩罚 | ≥ 0 | 派生 |

### 1. 事件驱动的关系变化（受知情范围门控）

仅对知情者应用变化，幂等（同一事件 ID 不重复结算，见 §Test 重复事件幂等）：

```
对每个 c ∈ event.知情者:
    rel[c→target][dim]' = clamp( rel[c→target][dim] + Δevent[dim], DIM_MIN, DIM_MAX )
```

- 同一行为被不同人以不同信息版本知晓时可产生不同变化（见 §Edge Cases）。
- **示例**：救援事件 Δ={恩义:+20}，知情者甲 → 甲对救援者恩义 +20；不知情的乙不变。

### 2. 关系记忆衰减

各维度按自身速率向中性值衰减（怨恨可设较慢衰减以表达长期记仇）：

```
rel[A→B][dim]' = rel[A→B][dim] − mem_decay[dim] × ( rel[A→B][dim] − DIM_NEUTRAL )
```

- **示例**：信任=60，DIM_NEUTRAL=0，mem_decay=0.05 → `60 − 0.05×60 = 57`。

### 3. 协作意愿评分（输出结构化响应）

由多维关系加权 + 风险惩罚，映射为结构化响应而非单一数值：

```
coop_score = Σ_dim w_coop[dim] × rel[executor→requester][dim] − risk_pen
response = 接受    if coop_score ≥ accept_threshold
         = 要求保证 if guarantee_threshold ≤ coop_score < accept_threshold
         = 反对    if reject_threshold ≤ coop_score < guarantee_threshold
         = 拒绝    if coop_score < reject_threshold
```

- 此 `coop_score` 归一化后即 GDD_005 `willingness` 公式中的 `relation_term`。
- **示例**：w={信任:0.5,敬重:0.3,恩义:0.2,怨恨:−0.4}，rel={信任:70,敬重:40,恩义:30,怨恨:10}，risk_pen=5 → `0.5×70+0.3×40+0.2×30−0.4×10−5 = 35+12+6−4−5 = 44` → 与各阈值比较得响应。

### 4. 接受后的执行稳定性（无随机翻令）

```
一旦 response = 接受 → 执行偏差仅来自 quality（GDD_005）、环境、已知风险
关系不在执行的每阶段重新掷骰改变命令
```

- 强烈反对者若被合法命令驱使，明确提出风险或事后关系后果，而非静默拒绝或随机抗命（见 §Edge Cases、§Main Rules）。

### 5. 授权有效性

```
authority_valid(grant) = (now ≤ grant.期限)
                       ∧ grant.授予者 仍具授予权限
                       ∧ ¬grant.已撤销
```

- 授予者失能/失势时，已有授权按其有效期与撤销规则处理（不自动全部失效，见 §Edge Cases）。
- 关系高**不能**绕过此判定凭空获得权限（见 §Main Rules）。

## Data Model

- `RelationshipState`：主体、对象、信任、敬重、恩义、怨恨、最后变化时间。
- `RelationshipEvent`：行为、参与者、知情者、维度变化、原因标签。
- `FactionMembership`：人物、派系/组织、立场、职责、公开程度。
- `AuthorityGrant`：授予者、接受者、权限范围、期限、可撤销规则。
- `CooperationAssessment`：响应、公开理由、要求、风险关注。

## Player Inputs

请求授权、协作、信息或资源；作出承诺；选择是否接受人物提出的条件；在事件中承担、解释或推卸责任。玩家不能直接增减关系值。

## System Outputs

命令权限、协作响应、信息共享范围、任务承诺、人物条件、关系变化事件、战后支持/问责和供军师/AI 使用的社会约束。

## Dependencies

依赖 GDD_005 人物、GDD_001 时间和事件知情信息。被城市、侦察、军议、战前准备、后果结算和 AI 消费。

## Edge Cases

- 正式命令合法但执行者强烈反对时，人物应明确提出风险或事后关系后果，而非静默拒绝。
- 授权者失能或失势时，已有授权按其有效期与撤销规则处理。
- 同一行为被不同人物以不同信息版本知晓时，可产生不同关系变化。
- 信任高但敬重低时可以分享私密信息，却质疑军事判断。

## Failure Cases

无权限请求、目标不在场、关系记录主体错误、重复事件或超范围授权会拒绝处理且不改状态。协作失败必须指出玩家已知的主要原因；隐藏原因只能显示为未知或推测。

## Balancing Parameters

各维度范围、事件变化量、衰减/记忆规则、协作权重、担保要求、授权期限、知情传播和战后问责影响。参数由事件与规则配置提供。

## UI Requirements

关系页分别显示方向和维度，以历史事件解释变化；请求协作时展示已知支持与阻力，不显示隐藏性格或秘密派系真值；不得用单一心形/好感条代替。

## AI Requirements

AI 使用自身已知的关系、职责和承诺作决策，遵守同一授权规则。敌将鲁莽可影响风险选择，但不能让其忽略明确权限和已接受承诺而任意行动。

## Save / Load Requirements

保存方向性关系、事件历史或可审计摘要、派系立场、授权与承诺；各阵营的关系认知单独保存，读档不能从真值自动补全未知关系。

## Test Requirements

测试方向不对称、多维组合、授权有效期、知情范围、重复事件幂等、协作响应、隐藏原因不泄露、接受后不随机翻令和存档 round-trip。

## MVP Scope

少量人物关系网、四个关系维度、有限职责授权、协作请求和战后问责。派系仅作为少量立场标签与支持来源。

## Future Scope

组织派系竞争、秘密、亲属师友、誓约、婚姻联盟、谣言传播、长期政治议程、官职任免和势力分裂。

## Open Questions

- 关系维度是否需要在 MVP 中全部连续数值化，还是使用等级加事件历史更易理解？
- 合法命令遭强烈反对时，执行者可否请求改派而不构成抗命？
- 战后问责由固定场景事件驱动，还是由通用责任归因规则生成？
