# GDD_007 — 情报与侦察

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

将世界真值与阵营认知分离，通过有成本、有时效、有暴露风险的侦察产生报告，让不完全信息成为可管理的战略条件，而不是隐藏随机。

## Player Fantasy

玩家像真实主将一样拼接斥候报告、地形观察和人物传闻，决定哪些情报值得相信、还缺什么，以及是否冒险行动。

## Core Loop

识别情报缺口 → 选择目标、方法与执行者 → 承担时间/人员/暴露成本 → 解析侦察 → 获得带来源、观察时间、范围和置信度的报告 → 比较或复核 → 用于军议与计划。

## Main Rules

- 世界真值、侦察观察、情报报告和阵营知识是四个不同层次，UI 只能读取阵营知识。
- 侦察任务必须指定对象、目标区域/主题、执行者、方法和持续时间；“侦察全部”不是合法命令。
- 结果由目标可接近性、执行者能力、天气/地形、敌方警戒、投入时间和确定性随机流解析。
- 报告包含一个或多个 `Claim`；每项声明记录值/范围、来源、观察时间、置信等级与时效。
- 置信度表示来源与观察方法的可靠性，不等同于系统公开的“真实概率”。
- 旧报告不会自动变成错误提示；其时效下降，并在新证据冲突时并列展示。
- 失败、部分成功、误判和侦察暴露都必须可追踪。错误信息来自具体观察限制，不是任意假数据。
- 敌将性格、兵力、营地和补给线必须分别侦察或通过合理来源获知。

## Formulas

> 四层严格分离：世界真值 → 观察 → 报告 → 阵营知识，UI 只读阵营知识。置信度表达
> 来源可靠性，**不**等于系统公开的真实概率。误差来自具体观察限制，**不**是任意假数据。
> 同一随机流位置 + 同一输入 → 同一结果（固定种子可重放）。所有成本/风险/映射来自配置。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `access` | 目标可接近性 | [0, 1] | 地图/目标派生 |
| `exec_cap` | 执行者相关能力（如谋略/武勇） | [0, 1]（归一化） | GDD_005 |
| `env_mod` | 天气/地形对侦察的修正 | 乘数 0.5–1.5 | GDD_002/003 |
| `alert` | 敌方警戒强度 | [0, 1] | CounterIntelState |
| `t_invest` | 投入时段数 | 整数 ≥ 1 | 玩家命令 |
| `r` | 随机流归一化值 | [0, 1) | 确定性随机流 |
| `conf` | 声明置信度 | [0, 1] | 派生 |
| `age` | 报告自观察以来的时段数 | ≥ 0 | now − 观察时间 |

### 1. 侦察成功度

加权各因素，时间投入边际递减，敌方警戒为惩罚：

```
recon_score = clamp( w_a×access + w_c×exec_cap + w_t×(1 − e^(−t_invest/τ)) ) × env_mod − w_alert×alert
outcome = 成功      if recon_score ≥ success_threshold
        = 部分成功  if partial_threshold ≤ recon_score < success_threshold
        = 空手      if recon_score < partial_threshold
```

- 时间项 `1 − e^(−t_invest/τ)` 表达边际递减：投入越多收益增速放缓。
- **示例**：w={a:0.3,c:0.3,t:0.4}，access=0.6，exec_cap=0.8，t_invest=3，τ=2 → 时间项 `1−e^(−1.5)=0.777`；env_mod=1.0，w_alert×alert=0.1 → `(0.3×0.6+0.3×0.8+0.4×0.777)×1.0 − 0.1 = (0.18+0.24+0.311) − 0.1 = 0.631`。

### 2. 声明置信度（来源可靠性，非真实概率）

```
conf = base_conf(method) × exec_cap × (1 − alert_noise) × outcome_quality
```

- `conf` 表达"这条来源多可信",UI 不得将其呈现为系统真实概率(见 §Main Rules)。
- **示例**：base_conf(斥候)=0.7，exec_cap=0.8，alert_noise=0.1，outcome_quality=1.0 → `0.7×0.8×0.9×1.0 = 0.504`。

### 3. 估计区间误差（误差来自观察限制）

报告给出区间而非点值,区间宽度随置信度反向变化:

```
half_width = base_error × (1 − conf) × env_mod
claim_range = [ observed_value − half_width, observed_value + half_width ]
```

- 真值落在区间内是**倾向**而非保证；低置信 → 宽区间。误差是观察限制的确定性函数,非随机假数据。
- **示例**：observed=5000 兵，base_error=4000，conf=0.5，env_mod=1.0 → half_width=`4000×0.5=2000` → 区间 [3000, 7000]。

### 4. 时效衰减（旧报告不自动变错，只降时效）

```
freshness(age) = clamp( 1 − age / ttl(topic), 0, 1 )
effective_conf = conf × freshness(age)
```

- 不同主题 `ttl` 不同（兵力变化快、地形几乎不变）。时效降为 0 不删除报告,新证据冲突时并列展示(见 §Main Rules、§Edge Cases)。
- **示例**：conf=0.7，age=4，ttl=10 → freshness=`1−0.4=0.6`，effective_conf=`0.7×0.6=0.42`。

### 5. 侦察暴露概率

```
P(exposed) = clamp( base_expose(method) + k_alert×alert − k_skill×exec_cap, 0, 1 )
exposed = ( r < P(exposed) )
```

- 用确定性随机流判定;暴露是可追踪 gameplay 结果(见 §Failure Cases)。
- **示例**：base_expose=0.2，k_alert=0.5，alert=0.4，k_skill=0.3，exec_cap=0.8 → `0.2+0.2−0.24=0.16`；r=0.1 < 0.16 → 暴露。

### 6. 冲突报告并列（不裁决真值）

```
当两份报告对同一主题 claim 冲突:
    都保留, 各自标注 effective_conf 与观察时间
    UI 提示矛盾, 不用隐藏真值替玩家裁决
```

- 满足 §Edge Cases"两份高置信报告冲突并列"与 §Main Rules"不用隐藏真值裁决"。

## Data Model

- `ReconTask`：任务 ID、目标、主题、执行者、方法、开始/结束时间、投入、状态。
- `Observation`：实际观察、地点、时间、可见范围、质量、是否被发现。
- `IntelReport`：报告 ID、来源、生成时间、声明集合、总体限制。
- `IntelClaim`：主题、估计值/区间、置信等级、观察时间、失效规则、来源链。
- `FactionKnowledge`：阵营接受/保留的报告与已确认事实，不覆盖冲突来源。
- `CounterIntelState`：区域警戒、诱饵/遮蔽标签、侦察暴露记录。

## Player Inputs

创建、取消或延长尚可调整的侦察任务；选择目标主题、执行者与方法；比较报告、标记关注或请求复核。玩家不能直接把推测标记为世界事实。

## System Outputs

观察和报告、置信/时效变化、知识投影、情报冲突、侦察暴露、执行者风险，以及供军议、地图、准备和 AI 使用的合法已知信息。

## Dependencies

依赖 GDD_001 时间、GDD_002 环境、GDD_003 地图、GDD_005 人物和 GDD_006 信息共享。向 GDD_008—010 与敌我 AI 提供知识。被 GDD_015（玩家对世界态势的认知受本系统情报限制）、GDD_016（敌方 AI 的唯一敌情来源，AI 只读自身 FactionKnowledge、永不读真值）消费。

## Edge Cases

- 任务执行中目标移动时，报告记录最后观察位置和时间，不自动追踪。
- 两份高置信报告冲突时并列保留并提示矛盾，不用隐藏真值裁决给玩家。
- 执行者失能或路线中断时可返回部分观察。
- 玩家亲眼可见的公开事实仍需带观察时间，离开后会过时。

## Failure Cases

目标不可达、执行者无资格、任务冲突、权限不足或资源未承诺会拒绝任务。被发现、空手而归、观察错误和来源欺骗属于 gameplay 结果，必须标明玩家可知限制且不泄露真相。

## Balancing Parameters

任务耗时/成本、方法风险、能力影响、地形/天气修正、警戒强度、置信映射、时效衰减、报告范围误差、暴露概率流和复核收益。全部配置化。

## UI Requirements

严格区分事实、报告、推测与未知；每条声明可展开来源、观察时间和限制；地图显示最后已知位置而非实时真值；冲突报告并列；不能只用百分比或颜色表达可信度。

## AI Requirements

AI 拥有独立 `FactionKnowledge`，通过相同侦察流程获取玩家信息。AI 不读取世界真值或玩家未执行计划；人格与目标影响其愿意为情报投入的成本和风险。

## Save / Load Requirements

保存侦察任务、观察、报告、阵营知识、来源链、时效、警戒和随机流位置。世界真值与玩家知识分别序列化，加载不得交叉污染。

## Test Requirements

测试真值隔离、固定种子结果、报告时效、目标移动、冲突来源、部分成功、暴露、权限失败无副作用、AI 知识隔离和存档 round-trip。

## MVP Scope

侦察敌将倾向、兵力范围、营地状态、补给路线和关键地形；少量侦察方法；来源、时效、置信和暴露完整闭环。不做庞大间谍网。

## Future Scope

间谍、内应、审讯、谣言、欺骗、密码、长期情报网、多源融合工具和外交情报交换。

## Open Questions

- 玩家能否主动“采纳”某份报告作为计划依据，还是所有报告始终并列？
- MVP 需要真实的敌方诱饵/假营，还是先用遮蔽和误差验证不完全信息？
- 置信度使用三/四档文本等级还是同时给估计区间？
