# GDD_001 — 游戏时间

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

提供所有系统共享的权威时间轴，使侦察、行军、生产、消耗、天气、疲劳和战役窗口在同一因果顺序中推进。时间是资源与机会成本，不是等待动画。

## Player Fantasy

玩家能选择“何时行动、愿意等待多久、错过什么窗口”，并感到昼夜与期限改变了可行策略，而不是只看到回合数减少。

## Core Loop

查看当前时段与期限 → 在暂停的计划阶段安排行动 → 检查耗时和冲突 → 提交命令 → 按确定顺序解析 → 推进一个或多个世界时段 → 接收时间事件与新决策窗口。

## Main Rules

- 权威时间以 `WorldDay + DaySegment` 表示；MVP 的时段类型为黎明、白昼、黄昏、夜间，具体长度来自配置。
- 计划、阅读和界面操作不推进时间；只有已提交且被接受的行动、主动等待或系统结算能推进。
- 战役使用嵌套的 `BattlePhase`。配置规定每个世界时段可容纳的战役阶段预算；耗尽后跨入下一时段并触发天气、补给和疲劳结算。
- 同一时间点的事件按稳定优先级和稳定 ID 顺序解析，不依赖帧率或集合遍历顺序。
- 行动声明开始时段、持续时段和可中断性；取消已经开始的行动不会自动返还已消耗时间。
- 到达时段边界时，先完成当前原子阶段，再依次结算时间、环境、补给、城市与状态事件。

## Formulas

> 所有常量来自版本化节奏配置（`TimeConfig`），此处以变量表示，不硬编码。
> 时间单位统一为「时段」（segment），1 世界日 = `SEGMENTS_PER_DAY` 个时段。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `SEGMENTS_PER_DAY` | 每个世界日的时段数 | MVP 固定为 4（黎明/白昼/黄昏/夜间） | 配置 |
| `d` | 世界日序号 | 整数 ≥ 0 | 运行时状态 |
| `s` | 时段枚举序数 | 0..SEGMENTS_PER_DAY-1 | 运行时状态 |
| `base_cost(a)` | 行动 a 的基础耗时 | 整数 ≥ 1（时段） | 配置 |
| `mod_terrain(a)` | 地形耗时修正（良道可加速） | 乘数，建议 0.5–2.0 | 配置 |
| `mod_weather(a)` | 天气耗时修正（**只减速，不加速**，与 GDD_003 对齐） | 乘数，建议 1.0–2.0 | 配置 |
| `PHASE_BUDGET` | 每个世界时段容纳的战役阶段数 | 整数 ≥ 1 | 配置 |
| `cancel_loss` | 取消已开始行动的时间损失比例 | 0.0–1.0 | 配置 |

### 1. 绝对时间索引（用于比较与排序）

将 `WorldDay + DaySegment` 线性化为单一可比整数：

```
T(d, s) = d × SEGMENTS_PER_DAY + s
```

- **用途**：判断两时间点先后、计算时间差、期限到期判定。
- **示例**：`SEGMENTS_PER_DAY = 4`，第 3 日黄昏（s=2）→ `T = 3×4 + 2 = 14`。

### 2. 行动耗时

```
cost(a) = ceil( base_cost(a) × mod_terrain(a) × mod_weather(a) )
```

- **约束**：结果向上取整且 ≥ 1，确保任何行动至少推进 1 时段。
- **示例**：base_cost=2，山地修正 1.5，雨天修正 1.3 → `ceil(2×1.5×1.3)=ceil(3.9)=4` 时段。

### 3. 行动完成时间

```
end_time = T(start_d, start_s) + cost(a)
```

- 完成时反线性化回 `(d, s)`：`d = end_time ÷ SEGMENTS_PER_DAY`（整除），`s = end_time mod SEGMENTS_PER_DAY`。
- **示例**：起于 T=14，cost=4 → end=18 → `d=18÷4=4，s=18 mod 4=2` → 第 4 日黄昏。

### 4. 战役阶段→世界时段的跨越

```
segments_consumed = floor( phases_elapsed ÷ PHASE_BUDGET )
```

- 每耗尽 `PHASE_BUDGET` 个战役阶段，跨入下一世界时段并触发天气/补给/疲劳结算。
- **示例**：PHASE_BUDGET=3，已进行 7 个战役阶段 → `floor(7÷3)=2` 个世界时段被消耗。

### 5. 期限剩余时间

```
remaining(deadline) = T(deadline_d, deadline_s) − T(now_d, now_s)
```

- `remaining ≤ 0` 即到期；`remaining ≤ warn_lead`（配置的提前量）时 UI 必须警告。
- **示例**：期限 T=20，当前 T=14，warn_lead=3 → remaining=6，未进入警告区。

### 6. 取消行动的时间损失

```
time_lost = ceil( elapsed × cancel_loss )
```

- `elapsed` 为取消时已消耗的时段数；已消耗时间按比例不返还。
- **示例**：已进行 4 时段，cancel_loss=0.5 → `ceil(4×0.5)=2` 时段不返还。

### 7. 稳定事件排序键（确定性保证）

同一时间点的事件按以下复合键升序解析，不依赖帧率或集合遍历顺序：

```
sort_key(e) = ( T(e.d, e.s), e.priority, e.stable_id )
```

- 三级排序保证同刻事件全序唯一，满足 §Test Requirements 的事件哈希一致性。
- **示例**：两事件均在 T=14，priority 分别为 1 和 2 → priority=1 先解析；若 priority 相同则按 stable_id 升序。

## Data Model

- `WorldTime`：日序号、时段枚举。
- `TimeWindow`：开始、结束、是否含端点。
- `TimedAction`：行动 ID、发起者、开始、持续、状态、可中断规则。
- `Deadline`：目标 ID、到期时间、到期后果类型。
- `BattleClock`：所属世界时段、当前阶段、阶段预算。
- `ScheduledEvent`：触发时间、稳定优先级、来源 ID、载荷引用。

## Player Inputs

提交有耗时的侦察、修整、移动、准备或等待命令；在允许的边界取消行动；确认跨越关键期限的计划。玩家不能直接编辑世界时间。

## System Outputs

时间推进结果、行动完成/中断事件、时段变化、日界变化、期限到达、需要环境/城市/补给结算的触发事件，以及供 UI 使用的时间预测。

## Dependencies

依赖版本化节奏配置和确定性事件排序。被季节天气、城市经济、侦察、战前准备、战斗、士气疲劳、后勤和存档消费。

## Edge Cases

- 多个行动在同一边界完成时按稳定顺序结算。
- 战斗跨越黄昏或午夜时，在原子阶段后更新环境而非阶段中途突变。
- 角色失能导致行动中断时，已用时间保留，未用资源按行动规则处理。
- 到期事件与玩家行动同刻发生时，优先级必须由配置契约明确且可预览。

## Failure Cases

非法过去时间、负持续时长、溢出日序号、重复事件 ID、无法解析的优先级或跨时段结算失败均拒绝提交，不产生部分推进。错过期限是可玩后果，不是技术错误。

## Balancing Parameters

每日日段定义、各行动基础耗时、地形/天气耗时修正、战役阶段预算、取消损失比例、期限长度与提示提前量。全部来自配置并标注单位。

## UI Requirements

常驻显示当前日与时段、下一边界、主要期限和计划耗时；时间轴区分预测与已提交状态；跨期限提交必须明确警告。暂停浏览不制造实时压力。

## AI Requirements

AI 与玩家共享时间和行动持续规则。AI 规划只能使用其认知中的期限与旅行时间，必须考虑机会成本，不得瞬时完成侦察、移动或准备。

## Save / Load Requirements

保存 `WorldTime`、战役阶段、进行中行动、调度事件、期限和稳定排序信息。读档后下一次推进必须产生与原运行相同的事件序列。

## Test Requirements

测试时段/日界推进、多个同步事件顺序、跨时段战斗、取消和中断、期限同刻冲突、非法输入无副作用、长时间溢出保护，以及同输入重复运行的事件哈希一致性。

## MVP Scope

有限天数、四类日段、少量定时行动、一个敌军压力期限和嵌套战役阶段。无月份 UI、节庆、年龄推进或长期历史事件调度。

## Future Scope

月份与年份、人物年龄、长期工程、历史事件窗口、旅行日程、季节农时和更复杂的并行行动调度。

## Open Questions

- 每个世界时段容纳多少战役阶段，才能兼顾战术选择与城市节奏？
- 同刻发生的期限和行动完成事件采用哪套玩家最易理解的优先级？
- 进行中行动在角色更换后允许接管，还是必须中断重启？
