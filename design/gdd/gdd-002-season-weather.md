# GDD_002 — 季节与天气

- 状态：Implemented
- **Status**: Implemented
- 范围：Vertical Slice

## System Purpose

生成可预测但不完全确定的环境状态，让天气、风向、昼夜和道路条件改变侦察、移动、补给与战斗窗口，同时保持结果可复现。

## Player Fantasy

玩家通过观察天象、听取预测和预留替代方案利用环境，而不是等待随机天气送出胜利。

## Core Loop

查看当前环境与预测 → 判断可信度和影响 → 安排行军、侦察或战役计划 → 时间推进时解析环境转移 → 条件改变道路、视野和行动 → 更新预测并应变。

## Main Rules

- 环境真值由场景气候、季节、前一状态、世界时段和显式随机流确定。
- 玩家只看到当前可观测环境和带范围/置信度的预测；预测不是未来真值副本。
- MVP 场景固定在一个季节窗口内，但季节配置仍影响昼夜、降水、雾和风的转移权重。
- 天气按世界时段边界变化，不在原子战役阶段中途切换。
- 风包含方向与强度；道路状态由近期降水、地形和恢复规则派生。
- 环境只修改明确消费者的条件，如视野、移动耗时、隐蔽、疲劳和补给损耗；不直接触发“计策成功”。
- 同一环境可同时产生利弊，例如雨降低能见度但恶化道路。

## Formulas

> 环境系统是「可预测但不完全确定」：转移由配置权重 + 显式确定性随机流驱动。
> 同一随机流位置 + 同一前态 → 同一结果，保证 §Test Requirements 的固定种子重放。
> 所有权重、速率来自版本化配置（`ClimateProfile`），不硬编码。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `W[from][to]` | 从天气 from 转移到 to 的权重 | ≥ 0 | 配置（转移表） |
| `r` | 随机流取出的归一化值 | [0, 1) | 确定性随机流 |
| `wet` | 道路湿滑度 | [0, 1] | 派生状态 |
| `rain_intensity` | 当前降水强度 | [0, 1] | EnvironmentState |
| `k_acc`、`k_rec` | 道路恶化/恢复速率 | 每时段，建议 0.0–1.0 | 配置 |
| `vis_base` | 天气类型基础能见度等级 | 0..VIS_MAX | 配置 |
| `forecast_acc` | 预测准确度 | [0, 1] | 配置 |
| `lead` | 预测提前的时段数 | 整数 ≥ 1 | 运行时 |

### 1. 天气转移（确定性加权选择）

在世界时段边界，从前态 `from` 按权重选下一态。先归一化为累积分布，再用随机流值定位：

```
P[to] = W[from][to] / Σ_t W[from][t]
next = first state where cumulative_sum(P[0..to]) > r
```

- **约束**：`Σ_t W[from][t] > 0`（无合法转移则阻止场景加载，见 §Failure Cases）。
- **示例**：from=晴，W={晴:6, 阴:3, 雨:1}，Σ=10 → P={0.6,0.3,0.1}，累积={0.6,0.9,1.0}；r=0.72 → 落在 (0.6,0.9] → next=阴。

### 2. 道路湿滑度积累与恢复

每时段按降水与恢复规则更新，clamp 到 [0,1]：

```
wet' = clamp( wet + k_acc × rain_intensity − k_rec × (1 − rain_intensity), 0, 1 )
```

- 降水期 `wet` 上升，干燥期下降；地形可乘以区域系数（同天气不同道路，见 §Edge Cases）。
- **示例**：wet=0.4，rain_intensity=0.8，k_acc=0.5，k_rec=0.3 → `0.4 + 0.5×0.8 − 0.3×0.2 = 0.4+0.4−0.06 = 0.74`。

### 3. 有效能见度

```
visibility = clamp( vis_base − weather_penalty − night_penalty, 0, VIS_MAX )
```

- `weather_penalty`、`night_penalty` 来自配置（雨/雾、夜间时段）。
- **示例**：vis_base=5，雨 penalty=2，夜间 penalty=1 → visibility=2。

### 4. 风效果门控

```
wind_effect = (wind_strength > 0) ? direction_effect(wind_dir) : none
```

- 风力为零时方向仅作历史值，不产生方向效果（见 §Edge Cases 零风）。

### 5. 预测置信度衰减

预测越远，置信度越低：

```
confidence(lead) = clamp( forecast_acc^lead, conf_floor, 1 )
```

- 预测显示候选状态 + 此置信度，**不等于**未来真值（满足 §Test 预测不泄露真值）。
- **示例**：forecast_acc=0.8，lead=3，conf_floor=0.1 → `0.8^3 = 0.512`。

### 6. 环境修正应用于消费者

环境只产出具名修正，由各消费者（移动/侦察/隐蔽/疲劳/补给）自行取用：

```
consumer_value = base_value × env_mod_multiplier[consumer] + env_mod_offset[consumer]
```

- 同一环境可同时产生利弊（如雨：能见度↓但隐蔽↑，道路↓），通过对不同消费者的不同修正实现（见 §Main Rules）。

## Data Model

- `ClimateProfile`：季节窗口与天气转移表引用。
- `EnvironmentState`：天气类型、强度、风向、风力、能见度等级、道路湿滑度。
- `WeatherTransitionContext`：世界时段、前态、地形区域、随机流位置。
- `Forecast`：覆盖时间窗、候选状态、置信度、来源、生成时间。
- `EnvironmentModifierSet`：对移动、侦察、隐蔽、疲劳和补给的具名修正。

## Player Inputs

查询当前环境和预测；选择等待、调整路线、夜间行动或准备替代方案。MVP 不允许玩家直接改变天气。

## System Outputs

环境状态变化事件、道路状态、可见度与风向投影、预测报告，以及传给地图、侦察、战前准备、战斗和后勤的具名修正。

## Dependencies

依赖 GDD_001 的世界时段、GDD_003 的地形标签和确定性随机源。被 GDD_003（世界地图通行耗时读已结算天气修正 mod_weather）、GDD_007、009、010、011、012 消费。

## Edge Cases

- 区域之间天气相同但道路状态因地形不同。
- 预测窗口跨越多个时段时必须逐段表达不确定性。
- 战斗跨越环境边界时先结束当前阶段，再应用新环境。
- 风力为零时保留方向只作为历史值，不产生方向效果。

## Failure Cases

缺少转移配置、非法概率权重、未知天气 ID、无效风向或无法映射的地形修正会阻止场景加载。恶劣天气导致行动失败属于 gameplay 结果，必须给出原因。

## Balancing Parameters

天气转移权重、状态持续倾向、预测准确度、风向变化权重、能见度、道路恶化/恢复速率、环境对移动/隐蔽/疲劳/补给的修正。全部版本化。

## UI Requirements

地图同时显示当前天气、风向、道路与能见度；预测用区间和置信度表示，不能与事实同样呈现；颜色以外使用纹理、箭头和文字。提交受天气影响的计划时列出已知主要影响。

## AI Requirements

AI 使用自身可得的当前观察和预测，不读取未来环境真值。人格可影响其对预测风险的容忍度，但不得改变天气规则。

## Save / Load Requirements

保存当前环境、道路派生状态或其权威输入、预测、气候配置指纹和随机流位置。读档后下一次转移必须一致。

## Test Requirements

测试转移权重校验、固定种子重放、时段边界、道路积累与恢复、预测不泄露真值、零风、跨时段战斗，以及每个消费者收到正确具名修正。

## MVP Scope

一个季节配置，少量天气状态，四/八向风，区域道路与能见度，单一阵营预测来源。重点验证夜袭、侦察、移动和补给决策。

## Future Scope

完整季节循环、洪水、干旱、积雪、河流水位、地方气候、多个预测来源、长期灾害和可干预的环境工程。

## Open Questions
