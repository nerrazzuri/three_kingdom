# GDD_008 — 军议

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

把玩家合法持有的情报、资源和人物观点整理为可质疑的条件化建议，帮助理解局势，但不替玩家选择或执行兵法。

## Player Fantasy

玩家召集不同性格和能力的军师、将领讨论局势，从相互补充或冲突的意见中形成自己的判断。

## Core Loop

选择议题与参与者 → 系统冻结合法知识快照 → 每位参与者提出观察、假设、所需条件、风险和缺失情报 → 玩家追问依据/比较分歧 → 玩家自行进入侦察或计划阶段。

## Main Rules

- 军议只读取召开时玩家阵营合法拥有的知识、公开城市/军队状态和参与者个人认知。
- 每条建议由观察、假设、条件、风险、缺失信息、依据和军师置信组成，不输出保证成功或隐藏真值。
- 建议可以描述“若敌将确实急躁且路线适合，可考虑诱敌”，不能自动选择人物、兵力、地点、时间并提交计划。
- 参与者能力影响其识别矛盾和遗漏的质量；性格、关系与立场影响关注重点和风险容忍。
- 多位参与者可以冲突。系统不生成全知的最终排名或“最佳方案”标记。
- 玩家追问只解释现有依据或提出新的情报需求，不能免费刷新出更优答案。
- 知识或资源变化后建议标记为过时，不能静默更新为新答案。

## Formulas

> 军议**只整理**合法知识为可质疑的条件化建议，**绝不**输出综合成功率、唯一推荐或
> 自动命令。能力影响识别矛盾/遗漏的质量，性格/关系影响关注重点，但系统不生成全知排名。
> 军师置信源自其可访问知识的置信（GDD_007），不凭空提升。所有权重来自配置，不写入生成方法。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `adv_cap` | 军师相关能力（谋略/治理等） | [0, 1]（归一化） | GDD_005 |
| `w_gap` | 能力对发现缺口/矛盾的权重 | ≥ 0 | 配置 |
| `claim_conf[i]` | 建议所依据第 i 条情报的有效置信 | [0, 1] | GDD_007 effective_conf |
| `trait_focus[t]` | 性格关注倾向强度 | −1..1 | GDD_005 |
| `snapshot_id` | 知识快照标识 | 稳定 ID | CouncilSession |

### 1. 矛盾/缺口识别质量

军师识别出的矛盾与缺失情报数量随能力提升,但上限为客观存在的数量:

```
detected = round( exist_count × clamp( w_gap × adv_cap, 0, 1 ) )
```

- 高能力军师接近发现全部客观矛盾;低能力可能遗漏。不会"无中生有"造出不存在的矛盾。
- **示例**：客观存在 4 处矛盾，w_gap=1.0，adv_cap=0.75 → `round(4×0.75)=3` → 识别出 3 处。

### 2. 军师置信（源自依据，不凭空提升）

单条建议的置信由其所引用情报的有效置信聚合(取最弱链路,因结论不强于最弱证据):

```
advice_conf = min_i( claim_conf[i] ) × adv_cap
```

- 取 min 体现"结论不强于最弱依据";两军师引用同一错误报告会形成一致但错误的高置信建议（见 §Edge Cases）——这是系统诚实暴露依据的结果,非缺陷。
- **示例**：依据置信 {0.6, 0.42, 0.7}，adv_cap=0.8 → `min(0.6,0.42,0.7)×0.8 = 0.42×0.8 = 0.336`。

### 3. 建议分歧检测（并列，不裁决）

```
conflict(a, b) = 互斥(a.结论, b.结论) ∧ (a, b 引用依据不同 ∨ 假设不同)
→ 生成 AdviceConflict, 并列保留, 不输出"哪个对"
```

- 系统标注冲突依据供玩家自行比较,绝不显示综合成功率或唯一推荐(见 §Main Rules、§UI）。

### 4. 建议过时标记（不静默更新）

```
stale(advice) = ( advice.snapshot_id ≠ current_knowledge_snapshot_id )
              ∨ ( 引用资源/情报自快照后已变化 )
```

- 过时建议标记但**不**静默重算为新答案;玩家须重新召开军议(见 §Main Rules)。读档后同快照建议文本/结构不变(见 §Save/Load）。

### 5. 追问成本（不免费刷新更优答案）

```
followup 只能: 解释现有依据 ∨ 生成新情报需求
followup 不能: 在不增加新情报的前提下产出更高置信的新建议
```

- 追问不重新随机生成;新洞察必须经由实际侦察(GDD_007)引入新情报后才可能出现(见 §Main Rules、§Test 追问不免费补全)。

## Data Model

- `CouncilSession`：议题、参与者、知识快照 ID、召开时间、状态。
- `AdvisorPerspective`：人物、可访问知识、能力贡献、性格/立场关注。
- `AdviceStatement`：观察、假设、所需条件、风险、缺失情报、依据引用、置信等级。
- `AdviceConflict`：互斥声明、涉及建议、冲突依据。
- `FollowUpQuestion`：问题类型、目标声明、回答或新情报需求。

## Player Inputs

选择议题和合法参与者；请求提出意见、解释依据、指出反对或列出缺失情报；固定关注建议。玩家必须另行创建侦察任务或战前计划。

## System Outputs

结构化建议、分歧、缺失情报列表、建议过时状态、参与者态度事件和跳转到相关信息/计划界面的只读上下文。

## Dependencies

依赖 GDD_005 人物、GDD_006 关系/信息共享、GDD_007 阵营知识，以及城市、地图、后勤和时间的只读投影。向 GDD_009 提供参考，不提供命令。

## Edge Cases

- 无军师在场时主将可自行整理已知事实，但没有额外洞察加成。
- 参与者持有玩家阵营尚未知的私人信息时，是否分享由关系与职责决定。
- 两位军师引用同一错误报告时可形成一致但错误的建议。
- 会后参与者失能不删除历史建议，只影响后续执行资格。

## Failure Cases

议题无效、参与者不在场/无权限、知识快照缺失或引用泄露隐藏真值时拒绝生成。建议质量低是 gameplay 状态，但系统必须仍提供其真实依据，不能伪造全知答案。

## Balancing Parameters

召开耗时、参与者上限、能力对矛盾发现/条件识别的权重、性格关注权重、关系对信息分享的影响、建议过时规则和追问成本。全部由版本化配置提供，不写入建议生成方法。

## UI Requirements

以人物分别呈现意见，明确区分观察与假设；每条建议可展开依据、条件、风险和缺口；冲突意见并列，不显示综合成功率或唯一推荐；离开军议不能自动填充部署。

## AI Requirements

敌方不需要玩家式军议 UI，但其规划器必须使用同等知识、人物能力、性格和分歧约束。AI 不得通过“军师系统”访问全局真值。

## Save / Load Requirements

保存会话、知识快照引用、建议、依据、分歧、追问和过时状态。读档后建议文本/结构不得因重新随机生成而变化。

## Test Requirements

测试隐藏真值隔离、错误报告传播、不同人物观点、分歧、不在场/权限失败、过时标记、追问不免费补全信息、无自动命令以及存档一致性。

## MVP Scope

一个战前军议议题、少量参与者、若干结构化建议模板，覆盖假退伏击、断粮疲敌和守城待变的条件与风险。无自然语言自由对话系统。

## Future Scope

朝议、外交议事、长期政策辩论、秘密议程、派系游说、可回顾的建议声誉和更丰富的问答表达。

## Open Questions

- 建议模板应由规则条件生成，还是由数据化论证图组合以支持扩展？
