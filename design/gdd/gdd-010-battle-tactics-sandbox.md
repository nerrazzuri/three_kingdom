# GDD_010 — 兵法沙盒战斗

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

在离散战役阶段和区域/路线空间中解析基础命令、环境、情报、人物、士气、军纪与后勤的组合，让伏击、诱敌、断粮、坚守和夜袭作为条件链涌现，而不是技能按钮。

## Player Fantasy

玩家以弱势兵力靠侦察、准备、纪律、地形、时机和敌将性格创造胜机，并能复盘“为什么成立、哪里差点失败、付出了什么”。

## Core Loop

读取战场公开状态与报告 → 提交一个战役阶段的基础命令/确认预设条件命令 → 双方锁定命令 → 按稳定顺序解析移动、侦测、接触、交战、士气和后果 → 显示可知反馈 → 进入下一阶段、撤退或结束战役。

## 战区与杠杆框架（自由布阵的落地）

> **术语锁（防跑偏）**：本作“自由排兵布阵”指**布局条件**——在战区中调度部队的位置、隐蔽与时机以创造伏击/断粮/火攻等条件链，**不是**阵型坐标摆放、兵种克制数值表，也**不是**实时微操。计策仍是条件涌现，**不是**技能按钮（见 §Formulas 设计锁）。

### 战区（Theater）

一场战役在一张**战区**上展开：由 GDD_003 的区域/路线构成的有限空间，每个区域带**地形标签**（平原/丛林/山口/河道/城防/营寨等），区域间有路线与通行耗时。战区提供“布局”的物理基础——大本营与粮道有远近、丛林山口可藏伏、河道渡口可扼守。双方在战区中下基础命令（移动/坚守/接战/侦察/受控撤退/隐蔽），战术由这些命令在战区上**形成的条件**涌现。

### 六类杠杆（Levers）

战术不是被逐条编写的“玩法”，而是若干**杠杆**被命令连接后涌现、且**互相组合**的结果。新增一个杠杆会与已有招式相乘（火 × 夜 × 伏…），故“丰富玩法 = 加杠杆”而非“加按钮”。杠杆均数据驱动（ADR-0003），新条件链以配置加入，不改解析代码。

| 杠杆 | 来源系统 | 提供的战术可能 |
|---|---|---|
| 天气/元素 | GDD_002 | 火攻、水攻、雾沙掩护 |
| 地形/空间 | GDD_003 | 隘口扼守、分进合击、诱敌深入 |
| 时间/昼夜 | GDD_001 | 夜袭、疲兵 |
| 补给 | GDD_012 | 断粮、坚壁清野、长围 |
| 人心/关系 | GDD_006 / 007 | 离间、策反、攻心（护城河独门） |
| 工程/兵力 | GDD_004 / 009 | 强攻（蚁附）、围城器械 |

### 招式库（条件链，皆涌现，非按钮）

每条均为多条件**全部成立**时涌现，系统**事后**识别并打复盘标签，绝无同名执行按钮：

- **火攻**（招牌，核心幻想）：干燥(GDD_002) + 下风口 + 敌处易燃（营寨/粮船/连营） + 火源就位 + 敌未察觉。
- **水攻**：敌处低地河谷 + 上游可决堤 + 雨季水位 + 筑坝时间。
- **断粮**：调动牵走敌主力（§7 敌追击判定）+ 小队袭粮道 + 补给延迟传导(§8)。
- **假退伏击**：受控撤退保形(§6) + 敌追击条件满足(§7) + 伏兵链就位（认知差 §4）。
- **夜袭**：夜间 + 隐蔽成功 + 敌未察觉 + 军纪达标(§9)；夜为**条件**（视野/疲劳/侦测惩罚），非强制作息表。
- **隘口扼守**：占桥/渡口/山口 + 高 terr_mod + 敌必经此路 → 以寡敌众守势。
- **分进合击（钳形）**：分兵藏一路 + 时机同步 + 敌被正面牵制 → 侧翼突袭（时机错则被各个击破）。
- **诱敌深入 + 坚壁清野**：弃外围烧粮 + 敌补给线随推进拉长 + 拖入腹地 → 断粮叠疲劳。
- **离间/策反/攻心**（人心杠杆）：情报得知敌将不和/处境(GDD_006/007) + 伪书/条件 + 时机 → 敌战力折损或阵前倒戈。
- **长围**：合围断补给与外援 + 时间 → 守军崩溃（“守城待变”的进攻镜像）。
- **强攻（蚁附）**：器械 + 兵力优势 + 接受高伤亡（无技巧基准线，供巧胜作对照）。

## Main Rules

- 战斗是确定性 Domain 模拟：初始快照、配置指纹、随机种子与有序命令流相同，则事件和状态哈希相同。
- 权威空间使用 GDD_003 的区域/路线，不依赖实时坐标、帧率或动画碰撞。
- 玩家基础命令包括移动、坚守、接战、受控撤退、追击、袭扰、隐蔽待命、侦察与休整；命令表达动作，不保证结果。
- 每阶段按“命令验证 → 移动/接触 → 侦测与隐蔽 → 交战匹配 → 损耗与位置 → 士气/疲劳/军纪 → 触发与撤退 → 事件发布”的稳定管线解析。
- 侦测决定阵营是否知道目标存在；突然性来自双方认知差，而不是点击“伏兵”。
- 受控撤退是否保持队形由军纪、士气、疲劳、指挥和路线决定；敌方是否追击由其知识、目标、性格、军纪与风险评估决定。
- 补给线受阻先改变 GDD_012 的供给状态，随后按时段影响疲劳、士气和可用行动，不立即造成魔法式全军崩溃。
- 夜袭由夜间、移动、隐蔽、侦测、军纪和接触共同形成；天气可能帮助隐蔽也可能增加迷失/疲劳。
- 结果允许成功、部分成功、暴露、反制、撤退、溃散或僵持；失败必须生成合法继续状态。
- 任何“兵法名称”只用于复盘标签。系统可在事件后识别条件链，不提供同名无条件执行按钮。

## Formulas

> **设计锁（不可违反）**：兵法是条件组合，**绝不**是无条件技能按钮。以下公式中
> 任何战术（伏击/诱敌/断粮/夜袭）都是多个可观察条件**全部成立**时的涌现结果，
> 系统只在事件后识别并打**复盘标签**，绝无同名执行按钮。
> 战斗是确定性 Domain 模拟：相同初始快照 + 配置指纹 + 随机种子 + 有序命令流 → 相同事件与状态哈希。
> 随机流**只**在规则声明的检查点消费。所有曲线、权重、修正来自版本化配置。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `force` | 单位当前兵力 | ≥ 0 | BattleUnitState |
| `morale`、`fatigue`、`discipline` | 士气/疲劳/军纪 | [0, 1] | GDD_011 |
| `terr_mod`、`posture_mod` | 地形/姿态战斗修正 | 乘数 | 配置 |
| `support` | 友邻支援系数 | ≥ 0 | Engagement |
| `awareness[A→B]` | A 对 B 的侦测状态 | 未察觉/怀疑/确认 | DetectionState |
| `supply_state` | 补给状态 | [0, 1] | GDD_012 |
| `r` | 随机流归一化值（仅检查点消费） | [0, 1) | 确定性随机流 |

### 1. 确定性状态哈希（可重放契约）

```
state_hash(phase) = H( initial_snapshot ‖ config_fingerprint ‖ seed ‖ ordered_orders[0..phase] )
```

- 同输入 → 同哈希,满足 §Test 同种子重放与状态哈希一致(也是 GDD_013 重放校验基础)。

### 2. 阶段解析管线（稳定顺序）

每个战役阶段严格按此顺序结算,每步原子;异常回滚整个阶段:

```
命令验证 → 移动/接触 → 侦测与隐蔽 → 交战匹配 → 损耗与位置
        → 士气/疲劳/军纪 → 触发与撤退 → 事件发布
```

- 双方同入同区按稳定接触规则创建交战,不由点击先后决定(见 §Edge Cases)。

### 3. 有效战斗力

```
combat_power(u) = force(u) × morale × (1 − fatigue) × discipline × terr_mod × posture_mod × (1 + support)
```

- 任一软状态低下都按比例削弱战力——这是非战斗系统（侦察/后勤/关系）影响战争的传导路径。
- **示例**：force=2000，morale=0.8，fatigue=0.3，discipline=0.9，terr_mod=1.2（据险），posture_mod=1.0，support=0.2 → `2000×0.8×0.7×0.9×1.2×1.0×1.2 = 2000×0.8×0.7×0.9×1.44 ≈ 1451`。

### 4. 突然性（来自认知差，非"伏兵按钮"）

突然性是**双方侦测状态之差**的涌现,不是可点击动作:

```
surprise(A on B) = ( awareness[A→B] = 确认 ) ∧ ( awareness[B→A] = 未察觉 )
ambush_bonus = surprise ? config.ambush_multiplier : 1.0
```

- **伏击 = 隐蔽成功（B 未察觉 A）+ A 已确认 B + 接触触发** 三条件链,任一不成立则无伏击加成。复盘时若识别此链,打"伏兵"标签。
- **示例**：A 隐蔽未被侦测（awareness[B→A]=未察觉）且 A 已确认 B → surprise=真 → ambush_bonus=1.5。

### 5. 交战损耗（双向，比例曲线）

```
casualty(B) = combat_power(A) / ( combat_power(A) + combat_power(B) ) × casualty_curve × ambush_bonus(A) × force(B)
```

- 双方各按对方相对战力受损;casualty_curve 配置化控制烈度。守城/据险通过 terr_mod 体现。
- **示例**：cp(A)=1500，cp(B)=1000，casualty_curve=0.3，ambush_bonus=1.5，force(B)=1000 → `1500/2500 × 0.3 × 1.5 × 1000 = 0.6×0.3×1.5×1000 = 270`。

### 6. 受控撤退是否保持队形

```
keep_formation = ( discipline × command_quality × (1 − fatigue) × route_clear ≥ formation_threshold )
```

- 不保持则转为溃散(force 额外流失、士气骤降)。撤退路线被阻则转向/投降/突围/受困,不瞬移(见 §Edge Cases)。
- **示例**：discipline=0.7，command_quality=0.8，fatigue=0.4，route_clear=1.0，threshold=0.4 → `0.7×0.8×0.6×1.0=0.336 < 0.4` → 队形溃散。

### 7. 追击决策（敌方由条件决定，无强制中计脚本）

```
pursue = ( w_k×knows_target + w_g×goal_fit + w_p×personality_aggression
         − w_d×discipline_restraint − w_r×perceived_risk ) ≥ pursue_threshold
```

- 敌将鲁莽通过 `personality_aggression` 与较低 `discipline_restraint` 表现,但仍是条件判定——AI 可识破不追(见 §AI Requirements)。
- **诱敌成立 = 假退（受控撤退保形）+ 敌追击条件满足 + 伏兵链就位**,三者皆备方成,复盘打"诱敌"标签。
- **示例**：w 各 0.2，knows_target=1，goal_fit=1，aggression=0.8，discipline_restraint=0.5，perceived_risk=0.3 → `0.2+0.2+0.16−0.1−0.06=0.4`；threshold=0.35 → 追击。

### 8. 断粮的延迟传导（不立即魔法崩溃）

> **单一权威点（避免重复施加）**：`supply_state` 由 **GDD_012 §5** 持有并结算；
> 断粮对 `morale`/`fatigue` 的变更由 **GDD_011 §1/§2** 作为**唯一施加点**结算（幂等）；
> 本系统（GDD_010）**只读取** GDD_011 已结算的 `morale`/`fatigue`，**不**自行再减，杜绝同一饥饿惩罚被算两次。

```
（GDD_012 §5 持有）supply_state' = clamp( supply_state − cut_rate × segments_cut, 0, 1 )
（GDD_012 §5 触发）shortage_segments ≥ grace_period → 发出断粮后果事件
（GDD_011 §1/§2 应用）morale/fatigue ← 按该事件唯一施加（幂等，按时段逐步，非瞬时）
（GDD_010 读取）combat_power 使用 GDD_011 已结算的 morale/fatigue，不重复扣减
```

- 补给中断先改 GDD_012 供给状态,经断粮事件由 GDD_011 **按时段**结算士气/疲劳,本系统只消费结果,不立即全军崩溃(见 §Main Rules)。
- **断粮疲敌成立 = 切断补给线 + 持续若干时段 + 敌士气/疲劳跨阈值**的时间条件链。

### 9. 夜袭条件链

```
night_raid_effective = is_night ∧ stealth_success ∧ (awareness[defender→raider]=未察觉)
                     ∧ raider_discipline ≥ night_discipline_min
weather_effect = 帮助隐蔽（visibility↓）⊕ 增加迷失/疲劳（route_risk↑）   （见 GDD_002）
```

- 夜袭由夜间 + 移动 + 隐蔽 + 侦测 + 军纪 + 接触共同形成,非按钮;天气是双刃(见 §Main Rules)。

### 10. 因果复盘（不泄露敌方隐藏条件）

```
top_factors = sort_desc( CausalTrace by |影响| ) 取前 ≤5 且 可向该阵营公开者
```

- 战后优先展示不超过 5 个决定性因素;敌方隐藏条件（如其真实士气）不可公开,以"敌军判断"等可知形式表达(见 §UI、§Open Questions)。

## Data Model

- `BattleState`：战役 ID、阶段、参与方、区域状态、目标、结束条件、随机流。
- `BattleUnitState`：单位 ID、阵营、位置、兵力、姿态、士气、疲劳、军纪、补给、指挥者、认知状态。
- `BattleOrder`：基础动作、主体、目标、参数、公开触发、提交序号。
- `DetectionState`：观察方、目标、未察觉/怀疑/确认、来源与时间。
- `Engagement`：接触双方、地形、姿态、支援、阶段状态。
- `BattleEvent`：类型、公开范围、原因因子、状态变化与稳定序号。
- `CausalTrace`：决定性因素、来源系统、影响方向、是否可向该阵营公开。
- `BattleOutcome`：目标结果、伤亡、撤退、俘获、资源变化与跨系统后果包。

## Player Inputs

为有权限控制的单位提交基础命令、目标区域/单位、姿态和允许的撤退/追击限制；确认或取消尚未锁定的阶段命令；查看己方已知反馈。玩家不直接选择命中、恐慌或兵法成功。

## System Outputs

区域与单位状态、公开/隐藏战役事件、侦测更新、伤亡、士气/疲劳/军纪变化、补给中断、撤退/溃散、目标结果、因果复盘和跨系统后果包。

## Dependencies

依赖 GDD_001 时间、002 环境、003 地图、005 人物、006 关系/权限、007 情报、009 准备、011 士气疲劳与012 后勤。GDD_013 保存快照与命令流。

## Edge Cases

- 双方同时进入同一区域时按稳定接触规则创建交战，不由客户端先后点击决定。
- 隐蔽单位和友军同区时必须分别维护各阵营侦测状态。
- 指挥者失能后，单位使用既有命令、代理规则或停顿，不自动获得新计划。
- 撤退路线被阻断时可转向、投降、突围或受困，不能瞬移回城。
- 双方同时满足结束条件时按目标优先级生成复合结果，而不是强制单方胜利。
- 随机流只在规则声明的检查点消费，跳过不适用检查不能改变其他流的语义。

## Failure Cases

非法命令、目标不可见且命令类型不允许、权限不足、单位已失能或配置缺失会拒绝该方提交并保持阶段未锁定。战术失败、误判、被反伏击和失城是 gameplay 结果；解析异常必须回滚整个原子阶段。

## Balancing Parameters

阶段预算、移动/接触顺序、地形与姿态修正、侦测/隐蔽权重、伤亡曲线、支援规则、撤退与追击条件、军纪检查、夜间/天气修正、目标权重和因果显示阈值。全部配置化并带单位与范围。

## UI Requirements

区域地图显示己方已知位置、路线、天气、风向、补给、士气和疲劳；命令预览列出已知关键条件和风险，不显示隐藏真值或单一保证成功率；战后优先展示不超过五个决定性因素，可展开完整因果链。

## AI Requirements

AI 与玩家共享命令、侦测、移动、战斗和后果规则，只使用自身知识。敌将鲁莽通过追击阈值和风险权重表现，不获得强制中计脚本；AI 可识破、犹豫或反制，结果由条件决定。敌方决策架构（效用评分 / 不全知知识 / 性格 / 计划 / 记忆 + LLM 装饰层 + 确定性保证）详见 **GDD_016** 与 **ADR-0006**。

## Save / Load Requirements

保存完整战役快照、阶段锁定状态、命令、随机流位置、侦测、交战、事件序号和配置指纹。仅在原子阶段边界正式落盘；阶段中的保存请求排队到安全点。

## Test Requirements

建立守城、假退未追、假退追击但伏兵暴露、伏击成功、断粮延迟生效、夜袭迷失、撤退路线阻断和失城继续等 golden scenarios；验证权限、隐藏信息、原子回滚、同种子重放和状态哈希。

## MVP Scope

一张**可玩战区**（一座城 + 有限区域/地形）、少量单位、阶段制命令。验收目标：在战区中通过布局完成 **守城待变、断粮疲敌、假退伏击、火攻**，外加一个**人心杠杆**（离间或策反）与**夜袭**作为组合手段——以此验证“条件链 + 杠杆”框架能否**通用地长出并组合**新招。无兵法技能树、实时微操、复杂兵种克制或大型攻城器械模拟。

## Future Scope

多战区、水攻/河战、工程攻城战、分进合击与更多空间机动、信使延迟、联军协同、长期围城和更丰富的战役目标；杠杆库与招式库持续扩充（数据驱动加条件链，不改解析代码）。
