# GDD_012 — 后勤与补给

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

把城市库存通过实际路线和运输过程转化为部队可用补给，使护送、截断、负载、道路和时间决定军队能否持续行动。

## Player Fantasy

玩家能追踪粮从哪里来、何时到、哪里可能被截断，并通过保护己方路线或破坏敌方路线改变战局，而不是点击“断粮”立即降低敌军属性。

## Core Loop

查看库存、单位需求和路线 → 创建补给分配/运输/护送/袭扰计划 → 提交资源与单位 → 按时段移动运输并消耗 → 交付或被阻断 → 单位库存与供给状态变化 → 延迟影响疲劳、士气和行动。

## Main Rules

- 粮食由城市/营地库存、运输批次和单位携行库存三类权威持有者保存，转移必须守恒。
- 单位按世界时段结算需求；优先消耗携行库存，再根据可达且已交付的补给获取。
- 补给路线是 GDD_003 路线的有序链，受控制、容量、天气、道路、运输单位和敌方接触影响。
- 创建运输时从来源库存原子扣除/保留，生成具名 `SupplyConvoy`，不能同时留在城市与运输中。
- 路线被切断不会删除已经携带的粮食；缺粮先形成短缺和持续时间，再由 GDD_011 产生疲劳/士气后果。
- 袭扰命令针对运输、节点或路线控制，结果由侦察、接触和战斗规则解析，不存在“断粮成功”按钮。
- 敌方补给真值对玩家隐藏，只能通过侦察、缴获、观察和公开后果推测。
- 缴获、损毁和遗弃均生成资源账本事件。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `S_city`、`S_convoy`、`S_unit` | 三类持有者库存 | ≥ 0 | SupplyStock/Convoy/UnitProvisions |
| `demand(u)` | 单位每时段需求 | ≥ 0 | SupplyDemand |
| `carried(u)` | 单位携行量 | 0..cap_carry | UnitProvisions |
| `convoy_speed` | 运输速度 | 区间/时段 | 配置 |
| `loss_rate` | 在途天气/道路损耗率 | [0, 1]/时段 | 配置 |
| `shortage_segments` | 连续短缺累计时段 | ≥ 0 | UnitProvisions |

### 1. 全局守恒不变量

```
任意时刻: ΣS_city + ΣS_convoy + ΣS_unit + 已消耗 + 已损毁/缴获 = 初始总量
```

- 任何转移是一个持有者减、另一个加的原子事务;拨粮从来源原子扣除/保留(见 §Main Rules)。满足 §Test 资源守恒与重放哈希。

### 2. 单位需求结算（先携行后交付）

每世界时段结算,优先消耗携行,不足再取已交付补给:

```
from_carried = min(carried(u), demand(u))
remain = demand(u) − from_carried
from_delivered = min(reachable_delivered(u), remain)
shortage = remain − from_delivered
carried(u)' = carried(u) − from_carried + new_delivery
```

- **示例**：demand=100，carried=60，reachable_delivered=30 → from_carried=60，remain=40，from_delivered=30，shortage=10。

### 3. 运输到达时间（沿路线链）

```
eta = Σ_{route ∈ convoy.路线序列} route_time(route)   （route_time 见 GDD_003，含天气/负载修正）
```

- 运输单位移动按 GDD_001 时段推进;路线被切断不删除已携带粮(见 §Main Rules)。

### 4. 在途损耗

```
S_convoy' = clamp( S_convoy × (1 − loss_rate × env_mult), 0, S_convoy )
```

- 恶劣天气/差道路提升 `env_mult`,逐时段损耗。
- **示例**：S_convoy=500，loss_rate=0.05，env_mult=1.4（暴雨）→ `500×(1−0.07)=500×0.93=465`。

### 5. 短缺时长累积（喂给 GDD_011，延迟传导）

```
shortage_segments' = (shortage > 0) ? shortage_segments + 1 : 0
若 shortage_segments ≥ grace_period:
    → 触发 GDD_011 疲劳/士气后果（见 GDD_010 §8 断粮传导）
```

- 缺粮先累计时长 + 宽限期,**不**立即魔法减益(见 §UI 逐时段因果链)。袭扰只改变运输/路线,经此链间接断粮,无直接按钮。
- **示例**：grace_period=3，连续短缺 4 时段 → `4 ≥ 3` → 触发疲劳/士气惩罚。

### 6. 同刻遇袭的部分交付

```
若 交付与袭击同刻: 按稳定事件顺序结算
delivered = min( convoy_remaining_after_attack, demand_pending )
captured/destroyed = convoy_total − delivered − convoy_escaped
```

- 可形成部分交付(见 §Edge Cases);缴获/损毁/遗弃各生成账本事件(守恒)。

### 7. 路线容量争用

```
∀ route: Σ_{convoy on route} convoy_load ≤ route.容量
超出时: 按承诺顺序 ∨ 玩家设定优先级 分配（见 §Open Questions）
```

- 多运输争用同路线时按承诺顺序或优先级,不静默叠加(见 §Edge Cases)。

## Data Model

- `SupplyStock`：持有者、资源类型、数量、保留量。
- `SupplyDemand`：单位、每时段需求、优先级、下次结算。
- `SupplyRoute`：来源、目的地、路线序列、容量、已知风险、状态。
- `SupplyConvoy`：批次、数量、运输单位、位置、护送、状态。
- `UnitProvisions`：单位携行量、容量、短缺累计时段。
- `SupplyLedgerEvent`：转出、运输、交付、消耗、缴获、损毁及原因。
- `SupplyStatus`：充足、紧张、短缺等配置化等级及公开原因。

## Player Inputs

从城市拨粮、设定目的地与路线、分配运输/护送、改变尚可调整的路线、命令袭扰或缴获、决定遗弃物资。玩家不能直接设置敌军为断粮。

## System Outputs

库存和运输账本、预计到达、路线容量/风险、单位供给状态、短缺时长、运输接触、缴获/损毁，以及传给时间、城市、准备、战斗和士气疲劳的事件。

## Dependencies

依赖 GDD_001 时间、002 环境、003 地图、004 城市库存、005 人物/单位、007 情报和010 战斗接触。向 GDD_009—011 提供供给状态。

## Edge Cases

- 运输在交付同刻遭袭时按稳定事件顺序结算并可形成部分交付。
- 目的单位移动后，运输按已提交目的地或重定向规则行动，不自动追踪隐藏位置。
- 路线容量被多个运输争用时按承诺顺序或优先级分配。
- 单位解散、投降或溃散时携行物资进入遗弃/缴获流程。

## Failure Cases

库存不足、路线无效、容量超限、运输者不可用或权限不足会拒绝提交且不重复扣粮。运输迟到、被截、部分损毁和单位挨饿是 gameplay 后果，必须保留账本和继续状态。

## Balancing Parameters

单位需求、携行容量、运输容量/速度、路线吞吐、天气/道路损耗、短缺宽限、缴获/损毁规则、护送成本、重定向耗时和供给状态阈值。全部配置化并标注资源单位和合法范围。

## UI Requirements

己方显示来源—路线—运输—单位的完整账本和预计到达；敌方只显示报告；路线用容量、风险和状态编码；断粮影响显示为逐时段因果链，不能立即显示魔法式减益。

## AI Requirements

AI 维持显式库存、运输、路线和需求，不能生成无限粮。其袭扰基于自身侦察，需权衡城防、护送和运输风险；难度修正若存在必须可配置并公开类别。

## Save / Load Requirements

保存所有库存、保留量、需求、路线、在途运输、位置、账本、短缺累计和下一结算点。加载后资源总量与下一次消耗/交付必须一致且不重复。

## Test Requirements

测试资源守恒、拨粮原子性、路线容量、天气耗时、部分交付、同刻遇袭、重定向、移动目的地、缴获/损毁、短缺延迟、隐藏敌情和重放哈希。

## MVP Scope

粮食一种完整资源；城市、敌营、补给节点、单位携行与少量运输；一条主要敌补给线；护送、袭扰、交付、短缺和士气疲劳联动。

## Future Scope

军械、马匹、药品、军饷、多级仓库、河运、补给优先策略、运输承包、腐败、季节储备和多城网络。

## Open Questions

- 在途粮食应在来源账本中显示为转出，还是保留“已发运”追踪汇总？
- 补给路线容量冲突采用先承诺先占用，还是允许玩家设定优先级？
- 敌军补给状态的公开后果要延迟多久，才能既可推断又不泄露真值？
