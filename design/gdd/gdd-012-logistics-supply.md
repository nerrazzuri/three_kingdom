# GDD_012 — 后勤与补给

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

把城市库存通过实际路线和运输过程转化为部队可用补给，使护送、截断、负载、道路和时间决定军队能否持续行动。

## Player Fantasy

玩家能追踪粮从哪里来、何时到、哪里可能被截断，并通过保护己方路线或破坏敌方路线改变战局，而不是点击“断粮”立即降低敌军属性。

## Core Loop

查看库存、单位需求和路线 → 创建补给分配/运输/护送/袭扰计划 → 提交资源与单位 → 按时段移动运输并消耗 → 交付或被阻断 → 单位库存与供给状态变化 → 延迟影响疲劳、士气和行动。

## Main Rules

- 粮食由城市/营地库存、运输批次和单位携行库存三类权威持有者保存，转移必须守恒。
- 单位按世界时段结算需求；优先消耗携行库存，再根据可达且已交付的补给获取。
- 补给路线是 GDD_003 路线的有序链，受控制、容量、天气、道路、运输单位和敌方接触影响。
- 创建运输时从来源库存原子扣除/保留，生成具名 `SupplyConvoy`，不能同时留在城市与运输中。
- 路线被切断不会删除已经携带的粮食；缺粮先形成短缺和持续时间，再由 GDD_011 产生疲劳/士气后果。
- 袭扰命令针对运输、节点或路线控制，结果由侦察、接触和战斗规则解析，不存在“断粮成功”按钮。
- 敌方补给真值对玩家隐藏，只能通过侦察、缴获、观察和公开后果推测。
- 缴获、损毁和遗弃均生成资源账本事件。
- **外交受控入口（支柱4 落点）**：MVP 只暴露一个受控外交请求（求援/求粮/求时限）。外势力为静态背景，其响应由场景预设、玩家阵营声誉/关系（GDD_006）和承诺代价决定；外援**延迟交付、可背约/迟到、需付代价**，绝非点击即到的保证按钮。该入口至少改变 slice 的兵力、时间或后勤之一。

## Formulas

> 粮食由三类权威持有者保存：城市/营地库存、运输批次、单位携行。**转移必须守恒**——
> 同一批粮不能同时留在城市与运输中。断粮通过实际路线切断 + 时段累积传导，**绝无**
> "断粮成功"按钮；缺粮先形成短缺时长，再由 GDD_011 产生疲劳/士气后果。敌方补给真值
> 对玩家隐藏。所有需求、容量、损耗、阈值来自版本化配置。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `S_city`、`S_convoy`、`S_unit` | 三类持有者库存 | ≥ 0 | SupplyStock/Convoy/UnitProvisions |
| `demand(u)` | 单位每时段需求 | ≥ 0 | SupplyDemand |
| `carried(u)` | 单位携行量 | 0..cap_carry | UnitProvisions |
| `convoy_speed` | 运输速度 | 区间/时段 | 配置 |
| `loss_rate` | 在途天气/道路损耗率 | [0, 1]/时段 | 配置 |
| `shortage_segments` | 连续短缺累计时段 | ≥ 0 | UnitProvisions |

### 1. 全局守恒不变量

```
任意时刻: ΣS_city + ΣS_convoy + ΣS_unit + 已消耗 + 已损毁/缴获 = 初始总量
```

- 任何转移是一个持有者减、另一个加的原子事务;拨粮从来源原子扣除/保留(见 §Main Rules)。满足 §Test 资源守恒与重放哈希。

### 2. 单位需求结算（先携行后交付）

每世界时段结算,优先消耗携行,不足再取已交付补给:

```
from_carried = min(carried(u), demand(u))
remain = demand(u) − from_carried
from_delivered = min(reachable_delivered(u), remain)
shortage = remain − from_delivered
carried(u)' = carried(u) − from_carried + new_delivery
```

- **示例**：demand=100，carried=60，reachable_delivered=30 → from_carried=60，remain=40，from_delivered=30，shortage=10。

### 3. 运输到达时间（沿路线链）

```
eta = Σ_{route ∈ convoy.路线序列} route_time(route)   （route_time 见 GDD_003，含天气/负载修正）
```

- 运输单位移动按 GDD_001 时段推进;路线被切断不删除已携带粮(见 §Main Rules)。

### 4. 在途损耗

```
S_convoy' = clamp( S_convoy × (1 − loss_rate × env_mult), 0, S_convoy )
```

- 恶劣天气/差道路提升 `env_mult`,逐时段损耗。
- **示例**：S_convoy=500，loss_rate=0.05，env_mult=1.4（暴雨）→ `500×(1−0.07)=500×0.93=465`。

### 5. 短缺时长累积（喂给 GDD_011，延迟传导）

```
shortage_segments' = (shortage > 0) ? shortage_segments + 1 : 0
若 shortage_segments ≥ grace_period:
    → 触发 GDD_011 疲劳/士气后果（见 GDD_010 §8 断粮传导）
```

- 缺粮先累计时长 + 宽限期,**不**立即魔法减益(见 §UI 逐时段因果链)。袭扰只改变运输/路线,经此链间接断粮,无直接按钮。
- **权威分工**：本系统**只**持有 `supply_state`/`shortage_segments` 并**发出断粮后果事件**；morale/fatigue 的实际变更由 **GDD_011 §1/§2 唯一施加**（幂等），GDD_010 只读取——三处不重复计算（见 GDD_010 §8、GDD_011 §2）。
- **示例**：grace_period=3，连续短缺 4 时段 → `4 ≥ 3` → 发出断粮后果事件，交 GDD_011 结算士气/疲劳惩罚。

### 6. 同刻遇袭的部分交付

```
若 交付与袭击同刻: 按稳定事件顺序结算
delivered = min( convoy_remaining_after_attack, demand_pending )
captured/destroyed = convoy_total − delivered − convoy_escaped
```

- 可形成部分交付(见 §Edge Cases);缴获/损毁/遗弃各生成账本事件(守恒)。

### 7. 路线容量争用

```
∀ route: Σ_{convoy on route} convoy_load ≤ route.容量
超出时: 按承诺顺序 ∨ 玩家设定优先级 分配（见 §Open Questions）
```

- 多运输争用同路线时按承诺顺序或优先级,不静默叠加(见 §Edge Cases)。

### 8. 外交受控入口（支柱4 在 slice 的唯一落点）

> **范围锁**：MVP **只暴露一个受控外交入口**，**不**运行完整天下外交模拟（见 `systems-index.md` 权威状态表）。
> 外势力为**静态背景**（无完整 AI），其响应由**场景预设 + 玩家阵营声誉/关系（GDD_006）+ 承诺代价 + 确定性随机流**决定。
> **设计锁**：外援**绝非**点击即到的保证按钮——必须满足「延迟交付 + 条件判定 + 代价 + 可背约/迟到失败」。
> 同种子 + 同请求 + 同前态 → 同 `grant_score` 与同 `fulfilled`（满足 §Test 重放一致）。该入口至少改变 slice 的**兵力 / 时间 / 后勤之一**（满足支柱4 最低承诺）。

补充变量：

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `standing` | 玩家阵营对该外势力的声望/信誉 | [0, 1] | GDD_006 声誉 + 场景预设 |
| `base_grant` | 外势力基础响应倾向 | [0, 1] | 场景配置 |
| `pledge_cost` | 玩家承诺代价（资源/未来义务） | ≥ 0 | 玩家命令 |
| `dipl_pressure` | 敌方对该外势力的反向外交压力 | [0, 1] | 场景配置 |
| `commit_lead` | 请求到交付的时段数（外援不即时） | 整数 ≥ 1 | 场景配置 |
| `betray_risk` | 背约/迟到/缩水风险 | [0, 1] | 派生（standing/代价/战局） |
| `r` | 随机流归一化值（仅兑现检查点消费） | [0, 1) | 确定性随机流 |

**8.1 响应判定（确定性，输出结构化响应而非单值）**

```
grant_score = clamp( base_grant + w_s×standing + w_c×f(pledge_cost) − w_p×dipl_pressure, 0, 1 )
response = 接受    if grant_score ≥ accept_threshold
         = 附条件  if cond_threshold ≤ grant_score < accept_threshold
         = 拒绝    if grant_score < cond_threshold
```

- `f(pledge_cost)` 为配置化边际递减函数（加码代价收益放缓）。「附条件」返回须满足的额外承诺（见 §8.5）。
- **示例**：base_grant=0.3，w_s=0.4、standing=0.6，w_c=0.3、f(cost)=0.5，w_p=0.2、dipl_pressure=0.4 → `0.3+0.24+0.15−0.08=0.61`；accept=0.55 → 接受。

**8.2 交付时间（外援不即时，含时限）**

```
arrival_T = now_T + commit_lead          （沿 GDD_003 路线或场景预设入场点；T 见 GDD_001）
```

- 援军/补给在 `arrival_T` 才进入战局；玩家须在交付前承受压力（守城待变链的「外交时限」即此）。

**8.3 兑现 / 背约判定（确定性随机流，可被战局影响）**

```
fulfilled = ( r ≥ betray_risk ) ∧ (玩家未违反承诺前提) ∧ (交付路线未被永久切断)
若 ¬fulfilled → 援军不到 / 迟到 / 缩水，按 §Failure Cases 生成可解释后果
```

- 兑现用确定性随机流，背约/迟到是**可追踪 gameplay 结果**，非随机黑箱。交付路线若被敌永久切断则转为运输失败（接 §3 eta、§6 遇袭）。
- **示例**：betray_risk=0.2，r=0.31 ≥ 0.2 且未违约、路线通 → 兑现。

**8.4 对 slice 条件的作用（场景三选一）**

```
援军: 在 arrival_T 于预设入场区生成 reinforcement units      → 兵力↑（喂 GDD_010）
补给: 在 arrival_T 交付外部 SupplyConvoy                      → 后勤↑（喂 §1 守恒、§2 需求）
时限: 改变敌军压力 Deadline 的 remaining                       → 时间↑（喂 GDD_001 §5）
```

- 每个 slice 场景只启用一种作用，避免扩张为完整外交。

**8.5 代价兑付与背约后果（守恒，不凭空资源）**

```
接受时: pledge_cost 原子保留/扣减（资源经 GDD_004/012 守恒；未来义务记入 GDD_006 条约信誉）
背约方（外势力或玩家）→ 承担 GDD_006 关系/声誉后果（standing 下降、问责事件）
```

- 满足 §1 全局守恒：外援补给计入总量，不破坏不变量。玩家或外势力背约均有 GDD_006 可审计后果，无「白嫖援军」。

- `SupplyStock`：持有者、资源类型、数量、保留量。
- `SupplyDemand`：单位、每时段需求、优先级、下次结算。
- `SupplyRoute`：来源、目的地、路线序列、容量、已知风险、状态。
- `SupplyConvoy`：批次、数量、运输单位、位置、护送、状态。
- `UnitProvisions`：单位携行量、容量、短缺累计时段。
- `SupplyLedgerEvent`：转出、运输、交付、消耗、缴获、损毁及原因。
- `SupplyStatus`：充足、紧张、短缺等配置化等级及公开原因。
- `DiplomaticPledge`：请求者、外势力 ID、类型（援军/补给/时限）、承诺代价、响应（接受/附条件/拒绝）、`grant_score`、`commit_lead`、`arrival_T`、`betray_risk`、兑现状态、声誉后果引用。

## Player Inputs

从城市拨粮、设定目的地与路线、分配运输/护送、改变尚可调整的路线、命令袭扰或缴获、决定遗弃物资；通过受控外交入口发起一次求援/求粮/求时限请求并承诺代价。玩家不能直接设置敌军为断粮，也不能跳过外援的延迟、代价与背约风险强制即时到援。

## System Outputs

库存和运输账本、预计到达、路线容量/风险、单位供给状态、短缺时长、运输接触、缴获/损毁，以及传给时间、城市、准备、战斗和士气疲劳的事件。

## Dependencies

依赖 GDD_001 时间、002 环境、003 地图、004 城市库存、005 人物/单位、006 声誉/关系（外交受控入口）、007 情报和010 战斗接触。向 GDD_009—011 提供供给状态；外交受控入口向 GDD_010（援军兵力）、GDD_001（敌军压力时限）与本系统（外部补给）提供 slice 条件变更，并向 GDD_006 写回条约信誉/问责后果。亦被 GDD_016（敌方 AI 读自身补给态势、断粮压力作效用评分与多日计划）消费。

## Edge Cases

- 运输在交付同刻遭袭时按稳定事件顺序结算并可形成部分交付。
- 目的单位移动后，运输按已提交目的地或重定向规则行动，不自动追踪隐藏位置。
- 路线容量被多个运输争用时按承诺顺序或优先级分配。
- 单位解散、投降或溃散时携行物资进入遗弃/缴获流程。

## Failure Cases

库存不足、路线无效、容量超限、运输者不可用或权限不足会拒绝提交且不重复扣粮。运输迟到、被截、部分损毁和单位挨饿是 gameplay 后果，必须保留账本和继续状态。外交请求被拒、附条件未满足、外势力背约或外援迟到/缩水均为 gameplay 后果，必须给出玩家可知原因并写回 GDD_006 声誉影响；代价已承诺则按兑付规则处理，不凭空返还。

## Balancing Parameters

单位需求、携行容量、运输容量/速度、路线吞吐、天气/道路损耗、短缺宽限、缴获/损毁规则、护送成本、重定向耗时和供给状态阈值。外交受控入口：`base_grant`、声誉/代价权重（`w_s`/`w_c`/`w_p`）、`f(pledge_cost)` 曲线、接受/附条件阈值、`commit_lead`、`betray_risk` 曲线、外援规模/补给量/时限变更幅度。全部配置化并标注资源单位和合法范围。

## UI Requirements

己方显示来源—路线—运输—单位的完整账本和预计到达；敌方只显示报告；路线用容量、风险和状态编码；断粮影响显示为逐时段因果链，不能立即显示魔法式减益。

## AI Requirements

AI 维持显式库存、运输、路线和需求，不能生成无限粮。其袭扰基于自身侦察，需权衡城防、护送和运输风险；难度修正若存在必须可配置并公开类别。

## Save / Load Requirements

保存所有库存、保留量、需求、路线、在途运输、位置、账本、短缺累计和下一结算点。加载后资源总量与下一次消耗/交付必须一致且不重复。

## Test Requirements

测试资源守恒、拨粮原子性、路线容量、天气耗时、部分交付、同刻遇袭、重定向、移动目的地、缴获/损毁、短缺延迟、隐藏敌情和重放哈希。**外交受控入口**：同种子+同请求+同前态产生同 `grant_score` 与同 `fulfilled`（确定性）；外援延迟交付不即时；背约/迟到/缩水产生可解释后果且无凭空资源（守恒）；代价接受即扣/挂账；拒绝与附条件路径返回稳定结构化结果；至少一种外交行动可验证改变 slice 的兵力 OR 时间 OR 后勤（支柱4 验收）。

## MVP Scope

粮食一种完整资源；城市、敌营、补给节点、单位携行与少量运输；一条主要敌补给线；护送、袭扰、交付、短缺和士气疲劳联动。**一个受控外交入口**（求援/求粮/求时限三选一作用于 slice），外势力为静态背景、无完整天下外交模拟。

## Future Scope

军械、马匹、药品、军饷、多级仓库、河运、补给优先策略、运输承包、腐败、季节储备和多城网络。

## Open Questions
