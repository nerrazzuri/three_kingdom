# GDD_003 — 世界地图

- 状态：Draft
- **Status**: Draft
- 范围：Vertical Slice

## System Purpose

用有限的区域/路线拓扑承载位置、移动、视野、补给和战役条件，使空间选择可读且无需连续实时大地图。

## Player Fantasy

玩家像在军图上研判道路、要隘、林地与营寨，选择在哪里观察、隐藏、拦截和承受风险。

## Core Loop

阅读已知地图与情报图层 → 比较路线耗时、暴露和补给价值 → 下达移动/侦察/控制命令 → 按时段解析通行与接触 → 更新控制、可见信息和战役入口。

## Main Rules

- 地图是由 `Region` 节点与 `Route` 边构成的有向或双向图；视觉坐标不参与 Domain 结算。
- 区域拥有地形、容量、控制权、战略设施和隐蔽标签；路线拥有连接、耗时、通行条件、暴露和补给能力。
- 单位只处于区域、路线上的移动状态或战役实例中，不存在权威的像素坐标。
- 移动耗时由路线、单位、天气和负载共同决定，并通过 GDD_001 推进。
- 地图真值与阵营知识分离。己方领地和亲自观察可提供事实，敌区信息必须由侦察更新。
- 进入敌对占据或被拦截的区域会创建战役接触；不自动判定胜败。
- 容量冲突、狭窄路线和阻断会限制部署与补给，但数值由配置决定。

## Formulas

> 地图为区域/路线拓扑图，视觉坐标不参与 Domain 结算。所有耗时、容量、暴露参数
> 来自版本化地图/规则配置，不硬编码。寻路与路径耗时必须确定性、可重放。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `base_time(route)` | 路线基础通行耗时 | 整数 ≥ 1（时段） | 地图配置 |
| `mod_unit(u)` | 单位移动速度修正 | 乘数，建议 0.5–2.0 | 单位状态 |
| `mod_weather` | 天气通行修正（来自 GDD_002） | 乘数，建议 1.0–2.0 | 环境状态 |
| `mod_load(u)` | 负载耗时修正 | 乘数，≥ 1.0 | 单位状态 |
| `cap(region)` | 区域容量 | 整数 ≥ 0 | 地图配置 |
| `occ(region)` | 区域当前占用 | 整数 ≥ 0 | 运行时状态 |
| `expose(route)` | 路线暴露等级 | 0..EXPOSE_MAX | 地图配置 |

### 1. 路线通行耗时

```
route_time = ceil( base_time(route) × mod_unit(u) × mod_weather × mod_load(u) )
```

- **约束**：结果 ≥ 1，经 GDD_001 `cost(a)` 推进世界时段（与时间系统一致）。
- **示例**：base_time=3，mod_unit=1.0，mod_weather=1.3（雨），mod_load=1.2（重载）→ `ceil(3×1.0×1.3×1.2)=ceil(4.68)=5` 时段。

### 2. 最短路径耗时（确定性寻路）

路径总耗时为各路线耗时之和；多候选时取最小，平局按稳定 ID 序破除（保证可重放）：

```
path_cost(P) = Σ_{route ∈ P} route_time(route)
chosen = argmin path_cost，平局取 route 稳定 ID 字典序最小者
```

- **示例**：路径 A=[r1(2),r2(3)]→5，路径 B=[r3(4),r4(1)]→5，平局 → 取 ID 序在前者。

### 3. 区域容量门控

```
can_enter(region, n) = ( occ(region) + n ≤ cap(region) )
```

- 超容时部署提交失败或形成明确队列，不静默叠加（见 §Edge Cases）。
- **示例**：cap=5，occ=4，欲进 n=2 → `4+2=6 > 5` → 拒绝，返回结构化容量错误。

### 4. 路线暴露与拦截窗口

单位在路线上的暴露随通行时长累积，达阈值则进入可拦截窗口：

```
exposure_accum = expose(route) × segments_on_route
interceptable = ( exposure_accum ≥ intercept_threshold )
```

- **示例**：expose=2，已在路线 3 时段，intercept_threshold=5 → `2×3=6 ≥ 5` → 可被拦截。

### 5. 相向移动接触判定

两单位在同一路线相向行进且进度交叉时创建路线接触：

```
contact = (route 双向) ∧ (progress_A + progress_B ≥ 1.0)
```

- `progress` 为 [0,1] 的路线行进比例。交叉则按路线规则创建接触或退回最近区域（见 §Edge Cases）。
- **示例**：A 进度 0.6，B 进度 0.5 → `0.6+0.5=1.1 ≥ 1.0` → 创建路线接触。

### 6. 知识时效（事实 vs 推测 — 时效权威归 GDD_007）

> **权威分工**：报告/观察的时效与置信衰减由 **GDD_007 §4 `freshness(age)`/`ttl(topic)` 唯一持有**。
> 本系统的 `MapKnowledge` **只记录每条空间信息的「最后观察时间」`observed_time`**，事实 vs 推测的判定**复用 GDD_007 的降级判定**，不在本系统另立一套 TTL，避免两处时效机制冲突。

```
is_fact(map_info) = GDD_007.fresh_enough( map_info.topic, now − map_info.observed_time )
                  ∧ (map_info.source 可信)
```

- 地图主题（地形几乎不变、控制/兵力变化快）的 `ttl(topic)` 取自 GDD_007 配置；超时降级为推测，UI 须区分呈现（见 §Main Rules 真值/知识分离）。


- `WorldMap`：区域集合、路线集合、地图版本。
- `Region`：稳定 ID、地形标签、容量、设施、控制方、相邻路线。
- `Route`：稳定 ID、起终区域、方向性、基础耗时、通行规则、补给容量、暴露标签。
- `MapPosition`：区域位置或路线行进状态与进度。
- `MapKnowledge`：阵营已知地形、控制、单位传闻、观察时间与来源。
- `MapChange`：控制变化、路线阻断、设施受损。

## Player Inputs

选择目标区域和允许路线；提交移动、待命、侦察、破坏路线、护送或撤退等基础命令；切换事实/推测/补给/天气图层。玩家不能拖动单位跳过路线。

## System Outputs

路径候选、耗时与已知风险、移动进度、接触事件、区域控制变化、路线阻断、视野/知识更新和供后勤使用的可达网络。

## Dependencies

依赖 GDD_001 时间、GDD_002 环境和单位移动/负载信息。向 GDD_002（地形标签，供天气派生道路状态）、城市、情报、准备、战斗和后勤提供空间契约。（002↔003 为生产者/消费者双向关系——天气读地图地形标签、地图读天气通行修正，破环顺序见 systems-index §跨系统结算顺序）

## Edge Cases

- 两支敌军同时从相反方向进入同一路线时创建路线接触或按路线规则退回最近区域。
- 路线在单位行进中变为不可通行时，单位停留、折返或进入受困状态，不能瞬移。
- 控制权改变不自动揭示该区全部敌情。
- 容量不足时部署提交失败或形成明确队列，不能静默叠加。

## Failure Cases

孤立必需区域、悬空路线、重复稳定 ID、负耗时、非法容量或无法找到合法路径会返回结构化错误且不移动单位。被围或无路可撤是可玩状态。

## Balancing Parameters

路线基础耗时、容量、地形通行标签、暴露程度、隐蔽条件、补给容量、控制转换规则、侦察覆盖与拦截窗口。参数均由地图/规则配置提供。

## UI Requirements

清晰区分区域与路线、已知与推测、当前控制与最后观察；路径预览显示耗时、天气、补给和暴露因素；多图层不能只靠颜色编码；地图选择仅生成命令草稿。

## AI Requirements

AI 在自身 `MapKnowledge` 上寻路和评估风险，不读取玩家隐藏部署或未侦察路线状态。AI 必须遵守相同容量、耗时、阻断和接触规则。

## Save / Load Requirements

保存地图版本、控制与损伤变化、所有实体位置/行进状态、各阵营知识和进行中的接触。视觉坐标可重建，不进入权威存档。

## Test Requirements

测试图结构校验、有向路线、路径耗时、容量冲突、同步相向移动、动态阻断、知识不泄露、控制变化和同命令重放。性质测试确保实体位置始终合法。

## MVP Scope

一座小城、城外若干关键区域、敌营、补给节点和有限路线；地形只保留影响移动、隐蔽、视野或防御的标签。不做全中国地图与战略缩放模拟。

## Future Scope

多城市世界、河运与海运、道路建设、区域季节变化、大地图分层、长途旅行、动态聚落和更复杂的控制区。

## Open Questions

- 相向单位在路线中接触时应创建独立路线战场，还是归入最近区域？
- 区域容量是硬上限还是超过后产生拥挤惩罚？
- 玩家是否需要手动指定完整路径，还是允许选择目标后从多个可见候选中确认？
