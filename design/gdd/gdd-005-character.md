# GDD_005 — 人物

- 状态：Implemented
- **Status**: Implemented
- 范围：Vertical Slice

## System Purpose

建立持续存在的武将主体，使身份、能力、性格、职责、健康和名声影响其可知信息、可下命令与执行方式，并承接战争后果。

## Player Fantasy

玩家以一个有权限边界、有关系和会受伤失势的人物参与乱世；其他武将也是有判断与代价的合作者，而非可替换卡牌。

## Core Loop

理解人物能力与处境 → 分配职责和任务 → 人物基于能力、性格、关系与状态作出响应 → 行动产生表现和后果 → 更新健康、疲劳、名声、职责与人生状态。

## Main Rules

- 每个人物有稳定身份、能力域、性格倾向、职责、健康、疲劳、名声和当前任务。
- MVP 能力域为统御、武勇、谋略、治理、交涉；它们影响过程质量，不直接解锁无条件技能。
- 性格使用具名倾向，如风险偏好、纪律观、荣誉观与耐心；性格改变判断权重、意愿和后果，不提供任意战斗光环。
- 职责决定合法权限和责任范围；能力高不能绕过授权。
- 人物同一时段只能承担兼容任务；任务冲突由准备系统拒绝或要求明确替换。
- 失能、受伤或被俘会中断相关任务并生成替代/失败后果。
- 玩家角色也受相同规则约束；界面不能赋予其全知或超越身份的命令权。

## Formulas

> 能力影响过程质量与成功倾向，**不**直接解锁无条件技能；性格改变判断权重与意愿，
> **不**提供任意战斗光环。低能力/性格反对是可解释结果，绝不用随机"突然不听令"代替
> （见 §Failure Cases）。所有量表、权重、曲线来自版本化配置。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `cap[k]` | 能力域 k 的等级（统御/武勇/谋略/治理/交涉） | 0..CAP_MAX | CapabilitySet |
| `w_cap[k]` | 任务对能力 k 的权重 | ≥ 0，Σ=1 | 配置（任务档案） |
| `trait[t]` | 性格倾向 t 的强度（风险/纪律/荣誉/耐心） | −1..1 | PersonalityProfile |
| `w_trait[t]` | 该决策对性格 t 的权重 | 实数 | 配置 |
| `health_factor` | 健康状态对执行的折损 | [0, 1] | HealthState |
| `rep` | 名声值（带受众标签） | 可正可负 | ReputationRecord |
| `rep_decay` | 名声每时段衰减率 | [0, 1] | 配置 |

### 1. 能力贡献（任务过程质量）

加权能力得分，归一化到 [0,1] 作为过程质量基底：

```
quality = ( Σ_k w_cap[k] × cap[k] / CAP_MAX ) × health_factor
```

- **约束**：`Σ_k w_cap[k] = 1`；`quality ∈ [0,1]`。质量影响成功倾向/效率，不是成败开关。
- **示例**：谋略任务 w={谋略:0.7, 统御:0.3}，cap={谋略:80, 统御:60}，CAP_MAX=100，health_factor=0.9 → `(0.7×0.8 + 0.3×0.6)×0.9 = (0.56+0.18)×0.9 = 0.666`。

### 2. 执行意愿（性格 + 关系传导）

人物对命令的意愿由性格倾向与情境加权而成（关系部分见 GDD_006）：

```
willingness = clamp( base_will + Σ_t w_trait[t] × trait[t] + relation_term, 0, 1 )
```

- `willingness` 低不等于拒绝执行，而是降低质量或要求条件（可解释，非随机）。
- **示例**：base_will=0.5，高风险任务 w_trait={风险:0.4}，trait={风险:−0.5}（保守）→ `0.5 + 0.4×(−0.5) = 0.5−0.2 = 0.3` → 意愿偏低。

### 3. 健康折损因子

```
health_factor = health_level_multiplier[ HealthState.level ]
```

- 各健康等级对应配置乘数（健康=1.0，轻伤<1.0，失能=0）；失能中断相关任务（见 §Main Rules）。
- **示例**：轻伤等级乘数=0.7 → 该人物所有能力贡献 ×0.7。

### 4. 任务并发兼容性

同一时段只能承担兼容任务：

```
can_assign(c, task) = (health_factor > 0)
                    ∧ has_authority(c, task)        （职责权限，见 GDD_006）
                    ∧ no_conflict(c.current_tasks, task)
```

- 任一条件不满足 → 准备系统拒绝或要求显式替换（见 §Main Rules、§Failure Cases）。能力高不能绕过授权。

### 5. 名声变化与衰减

```
rep'(audience) = rep(audience) + Δrep_event − rep_decay × rep(audience)
```

- 名声按受众标签分别累积（见 §Open Questions 统一受众标签方案）；事件增减 + 每时段向 0 衰减。
- **示例**：rep=40，事件 Δ=+15，rep_decay=0.1 → `40+15−0.1×40 = 55−4 = 51`。

## Data Model

- `CharacterState`：ID、身份、能力集、性格集、职责、健康、疲劳引用、名声、位置、任务状态。
- `CapabilitySet`：具名能力与等级。
- `PersonalityProfile`：倾向 ID、强度、公开程度。
- `RoleAssignment`：职责、授予者、权限范围、有效时间。
- `HealthState`：状态等级、伤病、限制、恢复进度。
- `ReputationRecord`：受众范围、事由、强度、时间。

## Player Inputs

在权限内任命、派遣、召回、询问、休整或请求协作；选择玩家角色行动。玩家不能直接编辑能力、性格、忠诚或任务结果。

## System Outputs

行动资格、能力贡献、性格判断因素、任务占用、健康限制、人物事件、名声变化，以及供关系、军议、准备、战斗和 AI 使用的人物投影。

## Dependencies

依赖 GDD_001 时间、GDD_003 位置、GDD_006 关系/权限和 GDD_011 疲劳。向城市、情报、军议、准备、战斗、GDD_006 关系（人物为关系主体）与 GDD_011 士气疲劳（指挥者影响）提供执行者状态。（005↔006、005↔011 为生产者/消费者双向关系，破环顺序见 systems-index §跨系统结算顺序）亦被 GDD_014（生涯：人物身份变化写回）、GDD_016（敌将 PersonalityProfile 性格倾向）消费。

## Edge Cases

- 人物在任务开始后失去职责时，已提交命令按授权快照或撤销规则处理。
- 多个身份/职责冲突时必须有显式优先级，不能合并权限。
- 玩家角色受伤失能时进入代理、撤退或后果流程，而不是自动恢复。
- 未知敌将性格保存在真值层，玩家只能从情报报告获知推测。

## Failure Cases

人物不存在、失能、位置不合法、任务冲突或权限不足时拒绝分配。低能力导致质量下降、性格导致反对或风险偏好是可解释 gameplay 结果，不得用随机“突然不听令”代替。

## Balancing Parameters

能力量表与贡献曲线、性格权重、健康状态限制、恢复耗时、任务并发规则、名声增减与衰减、角色行动基础耗时。全部配置化。

## UI Requirements

人物页区分已知事实、玩家推断和隐藏信息；显示职责、权限、当前任务、健康、疲劳和关键性格线索；分配前显示冲突与主要影响，但不展示敌方真值。

## AI Requirements

AI 人物使用同一能力、性格、职责、健康和任务限制。人格影响方案评估和反应，不允许 AI 因剧情需要忽略失能、位置或权限。

## Save / Load Requirements

保存全部人物权威状态、任务、职责有效期、伤病、名声记录和性格真值；各阵营对人物的认知由情报系统另存，不能从人物真值重建玩家知识。

## Test Requirements

测试能力贡献边界、职责权限、任务冲突、失能中断、身份变更、玩家与 AI 同规则、敌将信息不泄露、状态 round-trip 和固定输入下人物判断一致。

## MVP Scope

少量关键人物；五个能力域；有限性格倾向；职责、健康、疲劳、名声和单一主要任务。年龄、婚姻、家族、完整生涯事件不进入 slice。

## Future Scope

年龄与死亡、教育成长、婚姻家族、官职体系、个人目标、长期创伤、流亡转投、继承和广泛可玩武将。

## Open Questions

- MVP 中玩家角色失能后采用临时代理，还是立即进入战后失败分支？
- 性格线索在初识人物时应公开到什么程度？
- 名声按势力/地方分别记录，还是先使用带受众标签的统一记录？
