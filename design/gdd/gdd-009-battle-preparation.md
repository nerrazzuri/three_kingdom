# GDD_009 — 战前准备

- 状态：Implemented
- **Status**: Implemented
- 范围：Vertical Slice

## System Purpose

把玩家的判断转化为明确的人员、部队、资源、位置、时机和撤退承诺，并在执行前发现硬冲突，而不替玩家补全兵法方案。

## Player Fantasy

玩家亲手安排谁守城、谁侦察、谁隐蔽、多少粮草随军、何时撤退，清楚知道每项准备占用了什么并承担机会成本。

## Core Loop

选择目标与已知情报 → 在草稿中分配人物、部队、资源、位置和基础命令 → 检查硬错误与风险警告 → 调整或保留风险 → 提交计划 → 原子承诺资源并推进执行 → 接收中断/接触结果。

## Main Rules

- `PlanDraft` 不修改权威 gameplay state；只有提交成功后生成 `CommittedPlan`。
- 计划由基础命令和承诺组成，不包含“伏兵成功”“断粮成功”等结果命令。
- MVP 基础准备包括部署、移动、待命、隐蔽、守备、袭扰、护送、修整、撤退策略和预备队保留。
- 每项命令必须指定执行者、单位/资源、目标、开始窗口和可观察触发条件；系统不自动选择缺失字段。
- 硬冲突（同一人物/单位同时占用、资源不足、位置不可达、无权限）阻止提交；软风险只警告，由玩家决定。
- 条件命令只能引用本方可观察事件，如“敌军进入区域”，不能读取敌方隐藏士气或真值性格。
- 提交时冻结相关配置与知识引用；执行时世界仍可变化，计划可能部分失效。
- 取消已承诺计划按行动状态处理资源和时间，不一键无损回滚世界后果。

## Formulas

> `PlanDraft` 不修改任何权威 state；只有提交全部通过验证后才原子生成 `CommittedPlan`。
> 计划只含基础命令与承诺，**不**含"伏兵成功""断粮成功"等结果命令。硬冲突阻止提交，
> 软风险只警告由玩家决定。系统**绝不**自动填补缺失字段或一键最佳部署。所有阈值来自配置。

### 变量定义

| 变量 | 含义 | 范围 / 单位 | 来源 |
|------|------|-----------|------|
| `need[res]` | 计划对资源 res 的总需求 | ≥ 0 | PreparedOrder 汇总 |
| `avail[res]` | 当前可承诺量 | ≥ 0 | GDD_004 available |
| `occupy(c, w)` | 人物/单位 c 在时间窗 w 的占用 | 区间集合 | Deployment |
| `expire_t` | 条件命令过期时间 | 时段（GDD_001 T） | Contingency |
| `cancel_loss` | 取消损失比例 | [0, 1] | 配置 |

### 1. 硬冲突门控（任一为真即拒绝整个提交）

```
hard_conflict = resource_short ∨ time_overlap ∨ unreachable ∨ no_authority ∨ has_cycle

其中:
  resource_short = ∃ res: need[res] > avail[res]
  time_overlap   = ∃ c: occupy(c, w1) ∩ occupy(c, w2) ≠ ∅   （同一人物/单位时间窗重叠）
  unreachable    = ∃ order: ¬path_exists(order.目标)          （GDD_003）
  no_authority   = ∃ order: ¬has_authority(order.执行者, order) （GDD_005/006）
```

- 任一硬冲突 → 拒绝整个提交,**不**产生任何占用(全有或全无,见 §Failure Cases)。
- **示例**：need{粮:600} > avail{粮:500} → resource_short=真 → 整单拒绝,粮草未扣。

### 2. 循环依赖检测

命令依赖图须为有向无环图（DAG）：

```
has_cycle = ¬is_DAG( order_dependency_graph )
```

- 两命令循环依赖时验证失败(见 §Edge Cases)。用拓扑排序检测;可排序即无环。
- **示例**：A 依赖 B 完成、B 依赖 A 完成 → 存在环 → 拒绝。

### 3. 原子资源承诺

通过验证后,一次性原子扣减,部分失败则全部回滚:

```
commit: ∀ res: avail[res] −= need[res]   （单一事务,要么全部成功要么全部不变）
```

- 满足 §Test 原子提交;加载后不重复扣资源(见 §Save/Load 幂等)。

### 4. 条件命令触发与过期

```
fire(contingency) = 可观察触发条件成立 ∧ (now ≤ expire_t)
若 now > expire_t 且未触发 → 释放占用 ∨ 转入预定后备行为
```

- 条件只能引用**本方可观察**事件（如"敌军进入区域"），**不能**读敌方隐藏士气/真值性格（见 §Main Rules）。
- **示例**：触发"敌入 R3"，expire_t=T20；T18 敌入 R3 → 触发；若至 T20 未入 → 释放或转后备。

### 5. 取消损失

```
resource_refund = committed_res × (1 − cancel_loss × execution_progress)
```

- 已执行进度越高,可退资源越少;不一键无损回滚世界后果(见 §Main Rules)。
- **示例**：committed=100，cancel_loss=0.5，progress=0.6 → refund=`100×(1−0.5×0.6)=100×0.7=70`。

### 6. 提交后情报变化（保持有效，只标记风险）

```
new_intel_contradicts(committed) → committed.risk_flag = 变化
committed 仍有效, 不自动重写
```

- 提交时冻结配置与知识引用;执行时世界仍可变,计划可能部分失效但不自动重算(见 §Edge Cases、§Main Rules)。

## Data Model

- `PlanDraft`：目标、命令草稿、资源分配、知识引用、版本。
- `PreparedOrder`：命令类型、执行者、单位、目标、时间窗、公开触发、撤销规则。
- `ResourceCommitment`：来源、类型、数量、用途、状态。
- `Deployment`：单位/人物、起始区域、姿态、隐蔽意图。
- `Contingency`：可观察触发、允许响应、过期条件。
- `PlanValidationResult`：硬错误、软警告、冲突引用、已知风险。
- `CommittedPlan`：不可变提交快照、执行状态和事件历史。

## Player Inputs

创建/编辑/删除草稿项；分配人物、单位、粮草与区域；设置公开触发和撤退条件；请求验证；确认警告并提交；在允许边界取消或重派。

## System Outputs

验证错误/警告、资源与任务占用、已承诺命令、预计耗时、执行事件、失效/中断原因，以及传给地图、时间、后勤和战斗的合法命令流。

## Dependencies

依赖 GDD_001—008 的时间、环境、地图、城市资源、人物、关系权限、情报和军议参考，并消费 GDD_011/012 的当前军队状态。向 GDD_010 提供命令与部署。被 GDD_019（出征时复用备战产出 `CommittedPlan` 派生攻城初始条件）消费。

## Edge Cases

- 提交后新情报推翻依据时，计划保持有效但标记风险变化，不自动重写。
- 条件永不触发时，命令在过期时间释放或转入预定后备行为。
- 执行者失能时，只中断其负责项；其他项按依赖图决定继续或失效。
- 两项命令循环依赖时验证失败。
- 资源在提交与执行间因更高优先级世界事件损失时，产生明确短缺与部分执行结果。

## Failure Cases

硬冲突、权限不足、非法隐藏条件、不可达目标、循环依赖或配置版本不匹配会拒绝整个提交且不产生占用。计划被识破、迟到或部分执行是 gameplay 失败，并保留原因链。

## Balancing Parameters

准备耗时、部署容量、隐蔽投入、命令复杂度限制、预备队成本、取消损失、触发窗口、重派耗时、各任务资源需求和警告阈值。全部配置化并声明单位、范围与版本。

## UI Requirements

必须有清晰的 Draft/Committed 区分；地图和人物/资源面板同步显示占用；验证结果区分错误与风险；系统不得提供一键最佳部署、自动填满空位或把军师建议直接转成完整计划。

## AI Requirements

AI 使用相同计划结构、资源承诺、公开触发和验证规则。AI 根据自身知识与性格生成方案，不读取玩家草稿；难度不允许绕过占用、耗时或权限。

## Save / Load Requirements

保存草稿（可选玩家便利状态）、已承诺计划、知识/配置引用、资源占用、触发状态、执行进度和历史。加载后不能重复扣资源或重复触发命令。

## Test Requirements

测试草稿无副作用、原子提交、人物/资源冲突、隐藏条件拒绝、循环依赖、提交后情报变化、执行者失能、资源损失、取消规则、重复加载幂等和 AI 同规则。

## MVP Scope

支持小城守军与少量外勤部队，覆盖部署、守备、移动、隐蔽、袭扰、修整、预备队、撤退策略和有限公开触发。不做通用脚本语言或自动排兵。

## Future Scope

多阶段战役计划、联军协同、信使延迟、复杂条件命令、计划模板分享、长期动员和多战区资源竞争。
