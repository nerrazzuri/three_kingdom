## 0. 约定与引用说明

本文件是项目级无障碍需求的**权威层**——它定义"达标条件"，而不重复 `design/ux/interaction-patterns.md` 中已落地的控件级约定。两份文档的关系如下：

- 交互模式库（P1–P12）是**实现级约定**，声明了每个模式"在这里执行哪些无障碍手段"。
- 本文件是**需求级声明**，定义为什么要这样做、目标分级是什么、验收标准是什么、哪些留 Future。

交叉引用格式：`→ P3`、`→ P12` 等，表示该需求的落实机制见对应模式。

## 1. 无障碍承诺层级

### 1.1 合规基线

| 维度 | 目标层级 | 说明 |
|------|---------|------|
| 视觉（对比度/颜色/文字） | **WCAG 2.1 AA** | 强制达标范围 |
| 运动/前庭（动画/闪烁） | **WCAG 2.1 AA（SC 2.3.1 + 2.3.3）** | 强制达标 |
| 键盘可达性 | **WCAG 2.1 AA（SC 2.1.1–2.1.2）** | 强制达标 |
| 认知负荷 | **WCAG 2.1 AA 相关项 + 本作自定义标准** | 强制达标 |

> **取舍说明**：WCAG 2.1 AAA 中部分条目（如 SC 1.4.6 增强对比度 7:1、SC 1.2.6 手语）在 PC 单机策略 RPG 的 MVP 阶段不具备实现条件，不作为本项目目标。AAA 中对本作有意义的条目已在下文各节单独标注为 Future。

### 1.2 适配范围声明

| 玩家群体 | 本作覆盖情况 |
|--------|-----------|

| 认知差异（工作记忆、注意力） | **覆盖**：第 5 节细化 |
| 听觉障碍 | **基础覆盖**：本作无强音频依赖，第 6 节声明 |
| 触控/移动端辅助技术 | **不适用**：本作 PC 专属，无触控 |

## 2. 视觉无障碍

### 2.1 色彩对比度

| 元素类型 | 最低对比度 | 适用标准 |
|--------|----------|--------|
| 正文、数值、标签文字 | **4.5 : 1** | SC 1.4.3 Contrast (Minimum) |
| 大号文字（≥18px 常规体 / ≥14px 粗体） | **3 : 1** | SC 1.4.3 |
| UI 控件边界、焦点指示器、图标 | **3 : 1** | SC 1.4.11 Non-text Contrast |
| 禁用态文字 | 豁免，但需有明确"禁用"标签/图标 | SC 1.4.3 Exception |

> 本作界面信息密度高（P1 四层标识、P2 多信号、P6 多维分列），仿古艺术风格常见低对比度纹理底。**规则**：装饰性纹理底图可豁免，但承载信息的文字层必须满足 4.5:1——UI Toolkit USS 中文字颜色须在艺术底板定稿后以对比度检测工具（例如 Colour Contrast Analyser）复核。

### 2.2 色盲安全

**需求来源**：→ P3（颜色外冗余编码），本节将其提升为可验证的项目级标准。

- 颜色**绝不**作为唯一信息载体（SC 1.4.1 Use of Color）——所有用颜色区分的状态必须附图标、纹理、形状或文字第二通道。
- 地图图层（天气/补给/控制/暴露）每层有独立符号/纹理体系，可单独关闭某图层后仍可读其余层。
- 势力色彩（玩家/敌方/中立）：须在以下两类模拟下均可辨识——
  - 红绿色盲（Protanopia / Deuteranopia）
  - 蓝黄色盲（Tritanopia）
- **MVP 实现方式**：提供**至少一套替代配色方案**（色盲安全调色板），可在设置中切换。不要求实时滤镜着色器（后者列 Future）。
- 验证工具：Coblis 或 Adobe Color Accessibility 模拟，QA 必须在全部色觉模拟下对关键界面截图检验。

> **取舍**：实时着色器滤镜（对现有美术全局染色）技术复杂度高，且在 Unity 6.3 URP 下需要额外 Post-processing Pass——列 Future。MVP 仅要求替代配色资产。

### 2.3 文字最小尺寸

| 界面类型 | 最小字号（1080p 参考分辨率） | 说明 |
|--------|------------------------|-----|
| 正文、数值、来源标注 | **16px**（优先 18px） | 本作策略 UI 文字密集，建议偏高 |
| 次要标注（P1 层级标签、P2 时效标记） | **13px**（不低于此值） | 低于此值需申请豁免并记录 |
| 悬浮提示（Tooltip） | **14px** | 不得低于正文的 80% |
| 图例 / 地图标注 | **12px**（仅图例；不承载决策信息可用此下限） | 承载决策信息的地图文字须≥14px |

> **1440p / 4K 适配**：UI Toolkit 须以物理像素或动态缩放保证上述字号在高 DPI 屏幕下等比放大，而非固定像素值缩水。

### 2.4 文本缩放（Text Scaling）

**这是本作高优先度需求**：策略 UI 文字密度极高，P1/P2/P5/P6 模式均大量使用标注文字。玩家需要能在不破坏界面布局的前提下放大阅读。

**MVP 必达**：

- 提供游戏内**文字大小滑条**，支持 80%、100%（默认）、125%、150% 四档。
- 在 150% 下，核心信息面板（P1 信息四层、P6 多维状态）须能重排（Reflowing），不出现横向溢出截断或文字叠加。
- P8 常驻时间状态条在 150% 下须保持完整可读，不溢出 HUD 边界。

**实现约束（Unity 6.3 UI Toolkit）**：

- 使用 UI Toolkit 的 `font-size` USS 变量体系，统一用相对单位（em / %）而非绝对 px——这样全局缩放时无需逐控件修改。
- 文字容器设 `flex-wrap: wrap` 或动态高度，禁止固定高度截断。
- **已知风险**：P5 渐进展开的折叠面板在缩放后可能超出屏幕可见区域——需在 125%/150% 下分别走一遍手工可用性测试。

**Future**：操作系统级 DPI 缩放自动跟随（SC 1.4.4 Resize Text，AAA 变体 SC 1.4.10 Reflow）。

### 2.5 焦点可见性（Focus Visible）

（与第 3 节输入无障碍重叠，此处从视觉角度声明）

- 键盘焦点指示器对比度：焦点轮廓与背景之间 ≥ **3:1**（SC 1.4.11 + SC 2.4.7 Focus Visible）。
- 焦点轮廓宽度：**≥ 2px** 实线，不得用虚线或单像素线替代。
- 焦点不得被装饰层、地图图层遮挡——z-order 须保证焦点轮廓永远在最顶层渲染。

## 3. 输入与运动无障碍

### 3.1 全键盘可达性

**基线来源**：`design/ux/interaction-patterns.md` 全局约束（Overview 段落）已声明"所有交互元素键盘可达、无仅悬停状态"。本节将其具体化为可验证条件。

**强制规则（MVP）**：

- 游戏内**所有可交互元素**（按钮、下拉、折叠面板、地图图层开关、P4 草稿提交、P9 确认对话框）均可通过 Tab / Shift+Tab / 方向键 / Enter / Esc 完成。
- **无 hover-only 状态**（SC 2.1.1 Keyboard）：所有 Tooltip、展开内容、悬浮信息在键盘聚焦时同样触发，而非仅在鼠标悬停时出现。
- Tab 顺序须与视觉布局的阅读顺序一致（SC 1.3.2 Meaningful Sequence），不允许 Tab 序与屏幕位置大幅错位。
- 键盘使用者不得掉入"焦点陷阱"（SC 2.1.2 No Keyboard Trap）——模态对话框（P9）允许焦点约束在对话框内，但须提供 Esc 退出路径。
- **验收测试**：QA 测试员以纯键盘（鼠标断开）完成核心循环（情报查看 → 计划草稿 → 提交 → 存档），不依赖鼠标不得出现卡死。

### 3.2 键位重映射

- 提供游戏内键位重映射界面，覆盖所有快捷键与常用操作（打开情报面板、跳过动画、暂停/推进时间、图层开关等）。
- 重映射数据持久化存档，切换存档档位不重置键位设置。
- 重映射界面须提示冲突检测（同一键被两个操作绑定时高亮警告，不静默覆盖）。
- **不强制要求**：单独的无障碍控制器配置文件——列 Future。

> **取舍**：键位重映射是 PC 策略游戏的行业基线期望，且本作键鼠为主，实现成本可控，列 MVP 必达。

### 3.3 无同时多键按压需求

- 任何游戏内操作不得要求玩家**同时按压三键及以上**（如 Ctrl+Shift+X）。
- 双键组合（如 Ctrl+Z 撤销）允许，但须提供对应的单次操作替代（界面按钮或菜单项）。
- 本作回合制，无时间压迫——天然不存在"同时按住多键应对实时事件"的设计需求，但开发时须注意快捷键不意外堆叠。

### 3.4 Gamepad 部分支持

**当前状态**：Gamepad 标注"部分"支持，具体边界尚未定义（→ 开放问题 OQ-02）。

**最低期望（MVP / 应达）**：

- 主菜单、设置、存档界面：Gamepad 可完整导航。
- 游戏内核心信息面板（P1/P5/P6）：Gamepad 方向键可在焦点项间移动，A/确认键可展开折叠项。
- Gamepad 与键鼠可随时**热切换**，不要求重启或重新进入界面。
- 焦点可见性要求与键盘相同（Gamepad 使用时焦点指示器须渲染）。

**非 MVP 目标（Future）**：

- 全局 Gamepad 专属导航地图（光标替代方案）。
- 自定义 Gamepad 按键映射。

### 3.5 可调交互参数

- **无强制双击速度依赖**：双击若作为交互方式，须有单击替代路径。
- **无按住时长硬编码**：若未来设计有"按住"确认类操作（如长按放弃草稿），须提供可调或在设置中提供"单击切换"替代模式——MVP 阶段若无此类设计则不适用，但作为设计约束提前声明。
- **无限时输入**：本作回合制，时间只在玩家推进时前进，天然满足 SC 2.2.1 Timing Adjustable——须在设计评审中确认无引入隐性计时器的新功能。

## 4. 运动与前庭无障碍

**需求来源**：→ P12（可跳过动画且不影响结算），本节将其提升为项目级标准并补充 WCAG 引用。

### 4.1 闪烁限制（SC 2.3.1 Three Flashes）

- **任何动画或视觉效果不得在 1 秒内出现超过 3 次闪烁**——这是绝对约束，无豁免（SC 2.3.1 Three Flashes or Below Threshold）。
- 战斗结算、天气变化等视觉反馈须通过闪烁频率审查。
- QA 须使用 PEAT（Photosensitive Epilepsy Analysis Tool）对战斗结算等高动态界面做静态截图序列检测。

### 4.2 Reduced Motion（减少动态）

- 提供全局 **reduced-motion** 开关（游戏内设置），关闭后：
  - 所有界面切换转场简化为即时淡入淡出或直接切换，不做平移/缩放/弹出动画。
  - P6 多维状态的数值跳变动画关闭，直接显示终值。
  - P8 时间轴滑动动效关闭。
  - 地图视角缩放动画（若有）关闭，直接到位。
- reduced-motion 开关须在进入游戏前可访问（主菜单设置，不要求通关后才能设置）。
- **自动跟随操作系统设置**（Windows 减少动画、macOS Reduce Motion）列 Future；MVP 阶段游戏内开关即可。

### 4.3 动画可跳过（→ P12）

- 所有状态变化动画须可通过空格/Esc/Enter 即时跳过，跳过后直接呈现终态（不影响 Domain 层结算，呼应 ADR-0004 表现层/逻辑层解耦）。
- 长于 3 秒的动画须在画面中显示"按任意键跳过"提示。
- 战果复盘中的因果链动画逐步展示可跳过，但跳过后须提供可回看入口（玩家可重新触发，不强制只看一遍）。

## 5. 认知无障碍

本作核心玩法建立在信息层次与决策深度之上。以下需求确保信息深度不演变为认知障碍。

### 5.1 信息分层与渐进展开（→ P1 / P5）

- **任何面板的默认视图不得超过 5 个主要信息块**（P5 原则）；细节通过展开/折叠提供。
- 展开/折叠状态在同一会话内记忆（玩家不需要每次都重新展开同一信息）。
- P1 四层标识（事实/报告/推测/未知）须用**图标+文字+排版**而非记忆型颜色区分——玩家无需记住"绿色=事实"即可读懂层级。

### 5.2 无依赖记忆的隐藏状态

- 游戏内不存在"你必须记住面板 A 的数值，然后到面板 B 核对"的**必要操作流程**——若跨面板对比有设计必要，须提供一键并排或关联跳转功能。
- P2 三信号（来源/时效/区间）须**内联显示**在当前上下文中，不需玩家跳转另一界面查询解释。
- 所有图标须提供文字标签或可访问的说明（悬停/聚焦后显示名称），禁止无说明的纯符号（SC 1.1.1 Non-text Content）。

### 5.3 可行动的错误与风险信息（→ P7 / P9）

- 命令被拒绝时，错误信息须说明**具体原因**（哪个前置条件未满足）及**可行动的下一步**（修改哪项参数可通过验证），不得只显示"操作失败"（呼应 SC 3.3.3 Error Suggestion，本作设计层已有此要求）。
- P9 确认对话框须写明后果，不得只问"确定？"。
- 错误与风险的**视觉区分靠图标+文字+位置**，不靠红绿颜色单一区分（→ P3）。

### 5.4 暂停与信息检视

- 回合制天然满足"时间不自动流逝"——需在设计评审中确认**任何阶段**（含结算动画播放中）玩家均可打开信息面板检视，不被锁定。
- 战果复盘须可在结算后**随时重新访问**（不要求记住结算时展示的信息），复盘入口在战后界面和历史记录中均可达。

### 5.5 教程与学习曲线

- 教程（若有）须可**随时重温**（帮助中心或系统说明）——本作规则体系复杂，不允许"仅首次运行时显示一次"的不可复现教程。
- P1/P2 的图标体系须在首次出现时提供内联说明，不依赖玩家已读过外部手册。

## 6. 听觉无障碍

本作的强信息依赖在视觉层（文字/图标/排版），音频对局势判断非必要。即便如此，须声明下列基线。

### 6.1 音频不得作为信息唯一载体（SC 1.4.2 / SC 1.1.1）

- **任何对游戏状态有影响的提示音（警告音、确认音、错误提示音）须有视觉等价**——图标变化、文字通知、或界面状态高亮。
- 举例：若存档失败有提示音，须同时在 UI 上显示错误文字（→ P9 可行动错误信息）。
- 本作弱环境音、弱叙事配音——但若未来加入**语音旁白**（军师语音、传令兵报告），须提供文字字幕等价，且字幕须满足 2.3 节字号要求。

### 6.2 音量分控

- 提供独立音量滑条：**总音量 / 音乐 / 音效 / 界面音效**（至少 3 类分离）。
- 各滑条可独立静音（包括将某类设为 0）。

### 6.3 单声道选项（Future）

- 单声道音频选项（适用于单侧助听器用户）列 Future，MVP 阶段不作要求——本作音频设计权重低，优先级相应降低。

## 7. 屏幕阅读器 / 辅助技术支持范围

### 7.1 Unity 6.3 UI Toolkit 现实能力声明

Unity 6.3 LTS 的 UI Toolkit 引入了 Accessibility（a11y）API（`PanelSettings.enableAccessibility`、`AccessibilityNode` 等），可将 UI 元素信息暴露给平台 AT（Windows Narrator、macOS VoiceOver）。但该 API 在 6.3 LTS 时仍属**有限支持**状态：

- 支持基本的元素名称、角色、状态（可聚焦/禁用/展开）暴露。
- 不支持复杂的动态语义树更新（等价 `aria-live` / `aria-owns`）。
- 对话框焦点捕获与 AT 虚拟光标的交互行为在跨平台下存在差异。

> **参考**：在实现前须查阅 `docs/engine-reference/unity/` 中的最新 UI Toolkit 可达性 API 文档，不得假设训练数据所知的 API 状态有效（见 `VERSION.md` Knowledge Gap Warning）。

### 7.2 MVP 可达目标（Best-Effort）

以下元素在 MVP 阶段**尽力提供** AT 可读标注（非阻塞发布条件，但须在 QA 中记录覆盖状态）：

- 主菜单按钮、设置界面控件：提供名称与角色标注。
- 模态对话框（P9）：焦点进入时 AT 可播报对话框标题与主要内容。
- 存档槽列表：每个槽位有可读的文字标注（档位名、日期、状态）。

### 7.3 非 MVP / Future 目标

以下列 Future（产品稳定后投入专项工程时间）：

- P1 信息四层面板的完整 AT 朗读（动态内容更新时通知 AT）。
- P5 展开/折叠的 AT 状态播报（等价 `aria-expanded`）。
- 地图元素的 AT 导航（战略地图的可达性本身需要专项设计，超出常规 UI 范畴）。
- 全屏阅读器模式（纯文本摘要面板，为低视力/全盲玩家提供）。

## 8. 需求分层与优先级

### 8.1 MVP 必达（发布阻塞条件）

## 9. 验收标准

以下 pass/fail 条件由 QA 可独立执行，不依赖主观判断：

| # | 验收条件 | 判定方法 | 严重级别 |
|---|---------|---------|---------|
| AC-01 | 对任意 5 个正文标签做对比度检测，全部 ≥ 4.5:1 | Colour Contrast Analyser，记录截图与数值 | BLOCKING |
| AC-02 | 使用 Coblis Deuteranopia 模拟，主地图上至少 3 支不同势力可通过图标/纹理区分（不依赖颜色） | 模拟截图人工判读 | BLOCKING |
| AC-03 | 断开鼠标，纯键盘完成：打开情报面板 → 展开某 P1 条目 → 切换 P6 某维度 → 提交一项计划 → 触发 P9 确认并取消。全程无焦点丢失 | QA 手工执行，记录测试日志 | BLOCKING |
| AC-04 | 打开文字缩放 150%，打开 P1 四层标识面板与 P6 多维状态面板，无文字被截断或溢出容器边界 | 截图目视检查 | BLOCKING |
| AC-05 | 开启 reduced-motion，触发一次结算动画，验证动画不播放（或仅淡入淡出），结算数值与关闭 reduced-motion 时相同 | 两次结算比对数值 + 目视检查 | BLOCKING |
| AC-06 | 在结算动画播放中按 Enter/Esc，动画即时终止，界面呈现终态数值 | QA 手工执行，计时动画停止时间 ≤ 0.5s | BLOCKING |
| AC-07 | 重映射"打开情报面板"为非默认键，重启游戏，验证映射持久，且原键不再触发该操作 | QA 手工执行 | HIGH |
| AC-08 | 在任意界面触发命令失败，错误消息须包含具体失败原因字段（非仅"操作失败"），且区分不靠颜色单一通道 | QA 代码/界面检查 | HIGH |
| AC-09 | 使用 Windows Narrator 朗读主菜单，所有按钮被朗读出可识别名称（无"未知元素"） | AT 测试，录音记录 | ADVISORY |
| AC-10 | 将所有音量滑条设为 0，触发存档失败，UI 上须出现文字错误消息（不依赖提示音） | QA 手工执行 | BLOCKING |

## 10. 开放问题

| # | 问题 | 影响范围 | 建议解决方式 |
|---|------|---------|------------|
| OQ-01 | Player Journey 缺失（`design/player-journey.md` 不存在）——各模式在玩家何种情绪/认知负荷下出现尚属假设，可能影响认知无障碍需求的优先级排序 | §5 认知无障碍 | 先以 MVP 核心流程（情报→计划→提交→复盘）为最小 journey 锚点，完整文档后续补充 |
| OQ-02 | Gamepad 部分支持的具体边界未定义——"部分"是指支持主菜单导航？还是支持游戏内 P5 折叠面板？不同边界决定 §3.4 的实现范围 | §3.4 输入 | 由设计与 UX 在第一个 Gamepad 故事卡前明确范围声明，再落地模式级 Gamepad 导航顺序 |
| OQ-03 | Unity 6.3 UI Toolkit Accessibility API 的实际能力须项目内实测验证——训练数据不覆盖此版本（见 VERSION.md），建议建立一个 Spike 故事卡（≤ 2 天）实测 AT 暴露能力，再确认 §7.2 的 MVP 覆盖列表 | §7 屏幕阅读器 | 开一张 Spike 技术调研故事卡，产出：UI Toolkit AT API 在 Win/macOS 下的实测能力矩阵 |
| OQ-04 | 色盲安全替代配色方案需要美术介入——调色板方案可能与仿古美术风格有张力，须由 Art Director 评审并确认哪些颜色可替换、哪些是品牌硬限制 | §2.2 色盲支持 | 在美术风格定稿前提出调色板约束清单，由美术总监出替代方案供 QA 验证 |
| OQ-05 | P2 区间可信度的视觉刻度在 150% 文字缩放下的排版行为未验证——区间刻度（如"8k–12k"配视觉条）在大字号下可能溢出或折行导致误读 | §2.4 文字缩放 × P2 | 在第一个可信度 UI 实现后，150% 下手工测试并截图存档，不达标走 Blocker 流程 |
| OQ-06 | 若未来加入语音旁白（军师语音、传令兵报告），字幕系统（字号、背景遮罩、位置）须单独设计——本文档仅声明"须提供字幕"，未定义字幕规格 | §6.1 听觉 | 加入语音资产前，由 UX Designer 与 Audio Director 协作出字幕样式规范，对齐 §2.3 字号要求 |
