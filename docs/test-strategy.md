# 测试策略初稿

## 质量目标

证明规则正确、失败安全、跨系统契约稳定、战斗可复现、存档可迁移，并让设计调整能快速发现回归。

## 测试层级

### Domain 单元测试

覆盖纯 C# 规则、值对象、不变量、状态转换、错误结果和 Domain Events。每个 public method 必须有正常、边界、非法输入与副作用检查；涉及随机时使用显式种子或受控随机流。

### Application 测试

验证 Command 权限、用例编排、原子提交、错误映射和只读 Query。失败命令不得留下部分状态。

### 契约测试

验证配置 schema、ID 引用、范围、单位、版本、存档序列化与迁移。对每个适配器使用同一组契约测试。

### 集成测试

覆盖 Presentation 之外的“Command → Domain → Event → Projection → Save”流程。Unity EditMode 测试验证资源适配，PlayMode 只验证 Scene 绑定、输入和表现集成。

### Scenario 与回放测试

以小城防御建立 golden scenarios：守城、伏兵暴露、断粮成功、假退失败、夜袭遇恶劣天气、战败延续。记录初始快照、配置指纹、种子、命令流与关键状态哈希。

## 确定性验证

- 同平台重复运行同输入必须得到相同事件序列和状态哈希。
- 支持的目标平台之间在关键结算上必须一致；若采用量化容差，必须由 ADR 明确。
- 改变无关显示设置不得影响 Domain 结果。
- 集合遍历、随机调用次序和时间源必须有回归保护。

## 跨系统属性

优先使用性质测试验证：资源不凭空产生、士气保持合法范围、命令失败不改状态、已死亡/失能角色不能执行命令、存档 round-trip 保持状态、隐藏真值不进入玩家投影。

## 失败与可玩性测试

不仅测试胜利。每个战役主要失败结果必须验证：状态合法、存在继续路径、后果可解释、不会自动毁档、不会泄露隐藏信息。

## 测试数据

- 测试 fixture 使用最小显式数据，不依赖生产资产默认值。
- 平衡配置和测试构造配置分离，避免数值调整导致无关契约测试脆弱。
- golden data 变更需要说明预期规则变化并经过审查。

## 发布门禁

Domain/Application 测试、配置校验、存档 round-trip、迁移测试、确定性回放和目标平台 smoke tests 必须通过。忽略测试、非确定性 flaky test 或未解释的 golden diff 均阻断发布。

## 当前阶段行动

在 method specs 中为每个 public method 明确测试矩阵；在 vertical slice 计划中定义首批 golden scenarios。当前不创建测试代码。
