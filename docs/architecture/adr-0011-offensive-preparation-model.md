# ADR-0011 — 多维确定性出征准备模型（Multi-Dimensional Offensive Preparation）

> **Status**: Accepted（2026-07-04；lean 内联裁定——扩既有 ADR-0004 闭合因果，复用 GDD_010 兵法条件枚举，无新违反禁则，story 可引用）
> **相关**: GDD_019 出征攻城 v2 · epic-029 · ADR-0004（确定性定点）· ADR-0010（占城归属）· GDD_002/006/007/008/010/012

## Context（背景）

GDD_019 v1 的出征"闭合因果"只有三维：`兵力 + 补给 + 已派生兵法条件`，`OffensiveSetupService.Derive` 把它单调映射为战力/士气。用户要求把出征做成**真实的多维决策**——"派谁挂帅、如何编成兵种、走哪条布势路线、择何时机天气"，而非点一个"攻城"按钮。

约束（须同时满足）：
1. **确定性**（ADR-0004）：权威路径整数/定点，无旁路随机，同输入同输出、可存档可复现。
2. **守设计锁**（GDD_010）：兵法=条件涌现，**不做兵种克制三角、不做坐标阵型、不做实时微操**。
3. **反全知**（GDD_007）：出征前敌情只经玩家知识投影；未侦察拿不到需知己知彼的条件。
4. **复用而非新造**：尽量复用现有 `TacticTag`/`TacticCondition`、`WeatherType`、`DaySegment`、僚属花名册。

## Decision（决策）

**D1 出征准备扩为六维，全部经确定性纯函数派生为 `OffensiveForce`（战力/士气/携入条件）。**
六维：兵力 D1 · 补给 D2 · 将领编成 D3 · 兵种编成 D4 · 布势路线 D5 · 时机/天气 D6，外加侦察情报门 D7（见 GDD_019 §4a）。`Derive` 保持**纯函数、无随机、整数/定点**。

**D2 布势路线 = 复用现有四条兵法链，非坐标阵型。**
`ApproachPlan { FrontalAssault | FeintLure | ProtractedSiege | NightRaid }` 一一对应 `TacticTag`（正面强攻无兵法）。路线**只声明目标条件集**；数据模型**无 position/grid/facing 字段**（负向不变量，AC-3c）。这是"布势=位置/隐蔽/时机的条件选择"的落法，杜绝退化为 SLG 摆阵。

**D3 兵种 = 杠杆，非克制三角。**
`TroopComposition`（各 `TroopType` 数，和 ≤ 兵力）只提供**匹配加成或 0**：匹配路线/地形的兵种给 `+FitBonus` 与解锁条件门（如骑兵份额→追击类条件），**绝无"被克减益"**（负向不变量，AC-3b）。`Derive` 无 A>B>C 克制表。

**D4 其余维度做成"条件成型的门"，而非独立数值加法。**
所选路线的每条目标条件，仅当其门（兵种份额 / 主将智略·统率 / 军师随军 / 时段 / 天气 / 已侦察）全通过才携入开战态（GDD_019 §5 F3 门表）。门不齐 → 条件不成型、不打标签（延续 TR-battle-002 负向不变量）。**"准备决定胜负"由此长成多维**：同样选诱敌，有骑兵+军师+侦察+隘口才真设伏得手，否则退化硬碰。

**D5 将领战斗属性为新增 Domain 值对象，与僚属好感解耦。**
`OffensiveGeneral`（`CharacterId` + 统率/武勇/智略 Q16.16∈[0,1] + `GeneralSpecialty`）。来源=玩家僚属花名册（GDD_014/006），但战斗属性是新维度（RetinueMember 只有好感）。副将贡献按 `decay^k` **递减**（护栏：防堆将碾压兵力，接 W5 反支柱）。

**D6 侦察情报门体现反全知。**
`Scouted` 布尔门 {守方未察觉, 伏兵突然性} 一类条件，未侦察并施加 `IntelBlindPenalty` 战力折扣。守方战力的**呈现**只经玩家知识投影（军师/目标视图显示估计值+时效，不显真值）；结算所用真实守备在 Application/场景层持有，不进类型层玩家可见面（AC-7）。

**D7 出征草稿为发起前临时态，不入权威存档。**
六维组装是发起前草稿（可反复改），`LaunchOffensive` 消费它一次性派生。授权/占城计数/归属/自立倾向仍入统一存档信封（ADR-0009/0010）；草稿读档后重组（AC-8 注）。

## Consequences（后果）

**正面**
- "准备决定胜负"从单调三维升为**六维条件涌现**，决策深度大增，仍确定性可测。
- 复用四条现有兵法链做路线 → **零新造兵法**，与 GDD_010 条件涌现结算/复盘识别天然一致。
- 兵种/阵型以"杠杆+路线"落法**结构性守住设计锁**（负向不变量可反射断言），杜绝滑向 SLG。
- 将领属性解耦好感 → 后续可接 GDD_005/006 人物成长而不动好感语义。

**负面 / 代价**
- Domain 表面变大（多个新值对象 + `Derive` 分支 + 更宽 Config）→ 需充分单测（每维单调/门/负向不变量）。
- 门槛系数多（§8 Tuning）→ 平衡面变大，需平衡验证用例守住 W5 反支柱（堆将 decay + 出征成本 + 单次功绩上限）。
- 天气门受限于现有 WeatherType（无风/旱 → 火攻不入 MVP），与既有 scope 一致，但"火攻路线"暂缺。

## Alternatives Considered（备选）

- **A（保持三维，只接 HUD 按钮）**：最省事，但用户明确要"派谁/编成/布势/时机"的真实决策 → 否决。
- **B（引入兵种克制三角 + 坐标阵型）**：直觉上"更像 SLG"，但**直接违反 GDD_010 设计锁** → 否决。
- **C（本决策：六维准备 + 路线复用兵法链 + 其余维做条件门 + 兵种做杠杆）**：兼顾决策深度、确定性、设计锁 → 采纳。
