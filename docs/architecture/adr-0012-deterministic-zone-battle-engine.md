# ADR-0012 — 确定性区域战斗引擎（Deterministic Zone Battle Engine）

> **Status**: Accepted（2026-07-04；lean 内联裁定——扩 GDD_010 执行层，复用 ADR-0004 确定性 + 现有 TacticCondition，无新违反禁则，story 可引用）
> **相关**: GDD_021 战场区域战斗 · GDD_010/011/019 · epic-031 · ADR-0004（确定性）· ADR-0006（敌AI，见 ADR-0013）· ADR-0011（多维准备）· ADR-0009（会话装配）

## Context（背景）

战斗是本游戏与其他三国演义游戏的核心差异点，须最完善。GDD_021 定"战斗=战场区域部署 + 回合制战中调整 + 敌方区域AI + 攻守统一"，替换竖切脚本战斗。核心架构问题：如何让一个**多区域、多回合、含玩家动态调整与敌方AI**的战斗，仍满足项目铁律——

1. **确定性**（ADR-0004）：整数/定点、无旁路随机，同种子+同指令序→同结果、可存档续战、可回放。
2. **守设计锁**（GDD_010）：区域非坐标格子、无兵种克制三角、兵法=条件涌现非按钮。
3. **攻守统一**：一套引擎两向复用。
4. **反全知**（GDD_007）：双向情报投影，隐蔽区不互见真值。

## Decision（决策）

**D1 战斗为确定性回合状态机，纯 Domain。**
`ZoneBattleState`（战场 + 各区交战态 + 支队 + 回合钟 + AI 记忆/种子）是权威态。一个回合 = 确定性纯函数变换：`Resolve(state, playerCommands, aiDecision) → state'`。无 `MonoBehaviour`/`UnityEngine`、无 float 于权威路径、无隐式随机（ADR-0002/0004）。

**D2 战场 = 命名区域图，非坐标网格。**
`BattleField` = `Zone` 集 + 邻接关系。Zone 载地形/条件禀赋/软容量/邻接。**数据模型无 position/grid/facing/坐标字段**（负向不变量，延续 ADR-0011 AC-3c）。MVP 固定 5 区模板，**预留场景注入自定义战场**（数据驱动，构建期不可变配置，ADR-0003）。

**D3 部署与调整 = 命令，经前置校验。**
`DeploymentCommand`（初始部署/调动/投预备/改姿态）走前置校验→稳定错误码→无部分写入（延续命令契约）。调动只限**相邻区 + 在途 N 回合**（不能瞬移），在途支队不参与当回合结算（失位代价）。这是"战中调整"的确定性骨架。

**D4 条件按区、按回合涌现（复用 GDD_010 §7 + TacticCondition）。**
各区维护累计回合计数（伏兵蓄势/断粮累时）与已成条件集；某区凑齐门（地形+兵种份额+将属性+时机/天气+已侦察+累计回合）→ 该区兵法成型 → 结算加成/一次性涌现冲击。门不齐不成型、不打标签（延续 TR-battle-002 负向不变量）。**无同名"计策按钮"**。复用 ADR-0011 的兵种杠杆/无克制。

**D5 回合结算确定性优先序。**
一回合内多区、双方同时结算的先后须固定：`守方判定 → 攻方判定 → 涌现触发 → 减员/士气写回`，各区按 `Zone` 枚举/id 规范序遍历。同时互破关键区按此优先序裁定（可复现）。

**D6 反全知双向投影。**
玩家对敌区域只见情报投影（估计/时效，GDD_007）；敌AI只见其 `AiWorldView`（ADR-0013）。隐蔽区（伏兵）未被对方侦破则不入对方可见面。类型层玩家可见结构不含敌真实兵力/部署。

**D7 攻守统一 + 会话装配。**
同一引擎，攻/守仅初始阵营角色不同。战斗由 Application（沿 ADR-0009 装配脊梁）从出征（GDD_019 六维准备为初始部署输入）或守城发起、纳入全局日界序与统一存档信封（ADR-0009/0005）。`ZoneBattleState` 存读档 round-trip 一致、确定性哈希、支持战中存档续战。

## Consequences（后果）

**正面**
- 战斗成为可玩核心且仍确定性可测（同种子+同指令序→同哈希），可回放、可存档续战。
- 区域图 + 条件涌现结构性守设计锁（无坐标/无克制，负向不变量可断言），杜绝滑向 SLG。
- 攻守一套引擎，维护面小；六维准备/占城C/士气/情报/天气全复用。

**负面 / 代价**
- Domain 表面大（战场/区域/支队/交战态/回合结算/命令）→ 需充分单测（确定性/门/调整合法性/优先序/存档）。
- 回合数×区域数的节奏须平衡（防冗长，§11）。
- 替换竖切脚本战斗 → 需迁移守城那局的战斗接线（一次性，见 epic-031 S6）。

## Alternatives Considered（备选）

- **A 保留一击结算**：省事但无"战中排兵布阵"，用户明确否决（战斗是核心差异点）。
- **B 坐标网格战棋**：直觉更像 SLG，但违反 GDD_010 设计锁、退化为兵种克制微操 → 否决。
- **C 本决策：命名区域图 + 回合确定性状态机 + 条件按区涌现** → 采纳。

## Engine Compatibility（引擎兼容）

纯 C# Domain，无 `UnityEngine`，`dotnet test`（NUnit）可验。定点 Q16.16、无 float 权威路径（ADR-0004）。Presentation 薄壳只读投影渲染区域态（Unity 6.3 LTS UI Toolkit）。战场自定义配置经构建期不可变配置管线（ADR-0003）。

## Compliance / Guardrails

- 纯函数回合变换、无随机、整数/定点（ADR-0004）；AI 随机走 ADR-0013/0006 注入流。
- 负向不变量（断言）：区域/支队/命令模型无坐标字段；无兵种克制减益（延续 ADR-0011）。
- 反全知：类型层玩家可见面不含敌真值；隐蔽区双向不互见。
- 命令前置校验、稳定错误码、无部分写入。
- 装配层不拥规则（ADR-0009）：战斗规则在 Domain，发起/存档编排在 Application。
