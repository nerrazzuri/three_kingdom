# UX 与无障碍基础

## UX 目标

玩家应能回答三个问题：我知道什么、我能改变什么、结果为什么发生。界面服务于不完全信息下的判断，不替玩家做决定，也不泄露隐藏真值。

## 信息架构原则

- 明确区分事实、推测、传闻和未知，并显示来源、时效与置信度。
- 重要状态展示当前值、变化趋势、主要来源和潜在后果。
- 命令提交前显示成本、耗时、已知前置条件、冲突和不可逆后果。
- 军师建议使用“观察 → 假设 → 所需条件 → 风险”结构，不标记唯一推荐答案。
- 战果复盘显示可公开的因果链；隐藏情报只在合理揭示后出现。

## 交互边界

UI 只提交 Commands 并读取 Queries/Projections。乐观动画不能先行改写权威状态；命令被拒绝时保留玩家输入上下文并提供可行动的错误信息。

## 认知负荷控制

- 渐进披露复杂度：先给决策摘要，再允许展开来源与公式。
- 同一界面优先呈现当前决策相关条件，避免全系统仪表盘。
- 批量与重复操作需要可撤销的计划阶段，但已提交且产生世界后果的命令按规则处理。
- 关键术语、单位和图标跨界面保持一致。

## 无障碍基线

- 不以颜色、声音、动画或悬停作为唯一信息渠道。
- 文本可缩放，关键界面支持重排并避免裁切。
- 提供键盘完整导航、清晰焦点、可重绑定输入和合理点击目标。
- 提供字幕、音量分类、减少动态、闪烁限制和动画速度选项。
- 图表和地图使用形状、纹理、标签与对比度冗余编码。
- 时间压力不依赖实时反应；暂停与检查信息不受惩罚。

## 战术可读性

地形、视野、天气、风向、补给线、士气和疲劳必须能分别查看，也能在提交计划时看到相关叠加。不得用单一“成功率”隐藏全部条件；可提供区间与关键因素，但保留未知和误报。

## 失败体验

失败界面先说明发生了什么和当前可继续状态，再提供撤退、求和、失城后整顿等合法路径。读档是玩家选择，不作为唯一默认解决方案。

## 测试要求

- 用键盘完成 slice 核心流程。
- 在常见色觉差异模拟下辨识势力和状态。
- 放大文本后无关键信息丢失。
- 新玩家能区分事实与推测，并解释军师建议不是自动命令。
- 玩家能从复盘中指出至少两个影响战果的条件。

## MVP 边界

建立功能性线框、信息词汇、状态编码和核心导航；不以装饰性仿古 UI 或完整美术皮肤替代可用性验证。
