# MVP 路线图 — 把已有系统组装成一局完整游戏（2026-06-23）

> 依据：`design/concept/mvp-scope.md`（权威 MVP 定义）。
> 核心判断：**MVP 的 10 项必须实现系统能力已全部在 Domain 完成并测试通过**；
> 剩余工作 = Application 编排 + Unity 接线 + 内容/场景 + 一局收口 + 质量验收。**基本不需要写新 Domain 系统。**

## 1. 阶段划分（建议顺序）

### Phase A — 一局闭环骨架（最高优先）
**目标**：让现有竖切变成「有目标、会赢会输」的一局。这是把技术切片变成游戏体验的关键，也最早暴露玩法问题。

- A1. **胜负与局终条件**（Application 编排，新增少量规则或复用 epic-007/008）：守住 N 日 = 胜 / 城破或粮尽 = 败 / 击退敌主力 = 胜。局终触发 epic-008 后果结算。
- A2. **受命与盘点屏**（player flow step 1）：开局展示守城目标、时间压力、城市初值、已知敌情、可用人物概览。
- A3. **局终结算屏**：胜/败/撤退/失城分支（消费 epic-008 OutcomeContinuation），败局保留有意义的继续路径（mvp-scope 验收：失败后非「立即读档」）。
- A4. 把「推进时段」与胜负条件联动（敌每日逼近/增援转化为时间压力）。

### Phase B — 接入剩余命令到表现层（逐条沿竖切接缝复制）
每条都按既有模式：Domain 服务 → Application 用例（SessionService/新协调器）→ 只读投影 → Presentation View → Unity 面板/按钮 → dotnet 测试 + batchmode 编译 + Play 签核。

- B1. **军师建议**（epic-005 Council + 既有 `CouncilView`）→ HUD 军师入口面板：条件化建议、缺失情报、定性置信、免责（无最优解高亮，P11）。
- B2. **战前准备/部署承诺**（epic-006 Preparation）→ 准备屏：分配人物/兵力/粮草/时间，硬冲突校验，原子提交（同一资源不双用）。
- B3. **战役时段命令**（epic-007 Battle）→ 战役 HUD：move/hold/raid/retreat 基础命令 + 确定性解析 + 条件链复盘标签（事后打标签，非执行按钮）。
- B4. **士气/疲劳/军纪**（epic-007 Cohesion + 既有 `CohesionView`）→ 战役期三维分列显示（P6 不合并）+ 阈值态（Steady/Wavering/Routed）。
- B5. **后果写回**（epic-008 Outcome）→ 局终把战果写回城市/关系/名声/人物，并在结算屏可读。
- B6. **外交受控入口**（epic-004 Diplomacy §8）→ 求援/求粮/求时限一个入口：延迟交付、信誉/关系代价、可背约（mvp-scope §简化但必须可见）。
- B7. **人物花名册**（epic-003 Characters/Relationships）→ 关键人物（主将/军师/外勤武将/敌将性格）展示 + 执行意愿影响命令。

### Phase C — 内容与场景（把 SliceScenario 扩成完整一战）
- C1. **完整场景「汜水小城守御战」**：1 座可失守小城 + 1 敌进攻集团 + 1 主补给线 + 有限战役区域/路线（接 epic-002 Map 拓扑）+ 少数天气/昼夜（接 epic-002 Environment）。
- C2. **关键人物数据**：主将/军师/可外勤武将/敌将性格差异（数据驱动，原创——遵守红线①零复制现有三国作品资产）。
- C3. **三条核心条件链在 production 可达成**：假退伏击 / 断粮疲敌 / 守城待变（+ 夜袭可组合）。每条满足 mvp-scope §条件链验收 6 条（至少一个非战斗条件、需主动投入、有暴露风险、不自动补齐、成败皆可解释、写回≥2 个非战斗状态）。
- C4. **数值平衡**：把 `SliceScenario` 占位数值替换为可达成「至少两条链以不同代价取胜」的平衡配置（数据驱动，不写进方法体）。

### Phase D — 质量、验收与打包
- D1. **MVP 验收指标实测**（mvp-scope §验收指标 + §可测试阈值）：两条链不同代价取胜、玩家能解释决定性因素、单变量改变情报/补给/时机/关系会改变可行方案、军师不输出完整组合、失败可继续、同种子同结果。
- D2. **Kill Criteria 监控**（mvp-scope §Kill Criteria）：退化为固定计策顺序 / 无法从可见信息理解战果 / 非战斗状态只改百分比 / 军师无价值 / 失败必读档 → 任一触发则停扩内容回设计。
- D3. **自动回放一致性**：事件序列 + 关键状态哈希一致性验证（接 epic-004/009 哈希 + 存档）。
- D4. **Windows build 打包验证**：非 Editor 环境跑通（确认 DLL 桥/输入/UIDocument 在 player build 正常）。
- D5. **ADVISORY 视觉/无障碍签核补全**：对比度实测、文本 150% 无溢出、色盲调色板、焦点环（art-bible §7 + 三屏 §12）。
- D6. **CI 首绿确认** + 清工程债（`tools/_unity_probe/` 删除、DLL 桥未来转 UPM/CI 注入）。

## 2. 明确不做（mvp-scope §明确排除，守住范围）

全中国大地图 / 完整历史剧本 / 全武将集；完整外交/家族/官僚/贸易/AI 扩张；大规模实时战场 / 美术量产 / Mod / 多人；计策技能树 / 自动布阵 / 一键最优军议；实时军阵微操 / 数百单位同屏 / 战场物理；超自然法术 / 无双清场 / 脱离后勤英雄单位。

> 用途：每次想加内容前对照本清单 + Kill Criteria，避免范围蔓延。

---

## 3. 关键路径建议

1. **先 Phase A**（闭环骨架）——最小可赢可输的一局，立刻暴露玩法问题，性价比最高。
2. **再 Phase B 按链路价值排序**：B1 军师 + B3 战役命令 + B4 士气 + B5 后果，是「侦察→军议→准备→交战→后果」主循环的骨干；B2 准备、B6 外交、B7 人物补全博弈深度。
3. **Phase C 内容**与 B 并行推进（场景/人物/平衡随接线逐步充实）。
4. **Phase D 验收**在一条完整链能玩通后即可开始小步验证，不必等全部接完。

## 4. 风险登记（承接回顾）

- 多数 Domain 未接表现层 = CD-C3 风险的剩余面 → 每接一条即 Play 签核消化。
- 内容/平衡是新工作量（Domain 不需新写，但场景/数值/人物要从零搭，且守原创红线）。
- DLL 桥手动重建易漏 → 接线频繁期考虑 D6 的 CI 注入前移。
